# First-time fix — user-reported bugs

A user-reported bug gets fixed once. The reporter never has to send a second screenshot for the same symptom.

History (June 2026, the Ankify note-type thread): a lifetime user bounced **two** "fixed" claims for the same bug. Both fixes were logically correct and unreachable — the 5-minute polling worker constructed `SyncNotionPageToRacUseCase` without the `templateOverridesProvider` argument, so the carefully fixed resolution chain never executed on the user's path (their cards are created by the poll, not by manual sync). Round three found it in minutes by doing two things first: pinning the user's actual runtime entry point, and reading their production rows. This rule makes those two things mandatory up front instead of round three.

| Requirement | Do instead |
| --- | --- |
| Do not diagnose a user-reported bug starting from the unit whose name matches the symptom ("note type wrong" → the note-type resolver). The resolver can be perfect and never called. | First pin the user's **runtime entry point**: which route, controller, job, or worker actually ran for them — manual sync vs the 5-minute poll vs webhook vs upload vs batch. Confirm with their prod rows (`last_synced_at`, job rows) or prod logs which trigger fired. Then trace **top-down from that entry point** to the symptom. |
| Do not declare a fix to a multi-site class done on green unit tests. A class constructed in more than one place (`SyncNotionPageToRacUseCase` has three construction sites) can be fixed at one site and broken at the rest; a missing optional constructor argument is invisible to unit tests **and** to tsc. | `grep -rn "new <ClassName>"` and confirm the changed argument/behavior is wired at **every** site — at minimum the site on the reporter's path. State in the PR body which sites were checked. |
| Do not write an inline adapter lambda at a call site that forwards a subset of a provider's arguments (`(owner, pageId) => repo.load(String(owner), pageId)`). When the signature later grows, the adapter silently drops the new argument and the feature is dead at that site. | One shared factory next to the type (e.g. `makeAnkifyTemplateOverridesProvider`), used by every construction site, with a test asserting full argument forwarding. |
| Do not reply "fixed" to a reporter on green tests — or even on merge. Two of two early "fixed" claims in the June thread were false at send time. | A "fixed" claim requires all of: merged, **deployed**, reachability confirmed on the reporter's own trigger path, and — when their data is inspectable — their prod rows or a prod log line showing the resolved behavior. Anything less, write what was actually done: "shipped a change; your next sync should show X — tell me if it doesn't." |
| Do not treat a "still broken after your fix" reply as a logic bug in the thing you just fixed. | It is a wiring/reachability bug until proven otherwise. Assume the fix never executed on their path; verify invocation first (prod log line, construction-site grep), and only then re-open the logic. |
| Do not skip reproducing with the reporter's shape of data. Their global settings, missing settings rows, database-vs-page id, or trigger type is usually the exact branch the tests never covered. | Read their rows (read-only prod psql) and write the regression test against that shape — the test for the June bug should have been "database child page, no settings rows, global custom". |
