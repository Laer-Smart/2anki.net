# i18n / localization rules

The web app ships in **10 locales** (added 2026-07): `en, de, es, fr, it, ja, nl, pl, pt, ru`. `en` is the source of truth. Per GA, only **German (~4.5%)**, **Portuguese (~1.5%)**, and **Spanish (~0.8%)** have real traffic; the rest are opportunistic, shipped behind the picker, and **pending native review** — the `language_changed` analytics event tells you which earn a human pass. Don't pay for professional translation until the data justifies it.

## How it works

- **Library:** `react-i18next` + `i18next-browser-languagedetector`, initialized in `web/src/lib/i18n/index.ts`.
- **Glob namespace loader:** `index.ts` auto-loads every `web/src/lib/i18n/locales/<lang>/<namespace>.json` via `import.meta.glob('./locales/*/*.json', { eager: true })`. To add a surface's strings, **drop a new `<namespace>.json` into each language folder** and reference `t('<namespace>:key')` — **no edit to `index.ts`.** This is the whole reason parallel translation work can land without colliding on one shared file.
- **Namespaces are per-area:** `common` (default + fallback namespace), plus `account`, `ankify`, `chat`, `search`, `tools`, `errors`, `landing`, `marketing`, `downloadsx`, `pricingtable`, `previews`, `accountx`, `chrome`, `upload`, … Grep `locales/en/` for the current set.
- **Detection order:** `localStorage` (key `2anki-language`) → `navigator`. `fallbackLng: 'en'`, `load: 'languageOnly'`, `nonExplicitSupportedLngs: true` (so `de-DE` → `de`). **Auto-detect is aggressive by design** — any `de-*` browser gets German automatically. That is the requested behavior, not a bug; a user surprised by it usually has that language first in `navigator.languages`. Language ≠ currency ≠ country: never key currency off the UI language.
- **Cross-device persistence:** a logged-in user's choice is stored in `users.language` and synced through the generic `PATCH /api/users/me/preferences` endpoint (the same one `theme` uses). Anonymous users persist via `localStorage` only. `scheduleSync` no-ops when not logged in.
- **Server-side surfaces:** emails localize by **recipient** language — `UsersRepository.getLanguageByEmail` → `getEmailStrings(lang)` in `src/services/EmailService/i18n/` (English fallback; one HTML file per template, copy injected via the existing `{{var}}` substitution — **no partials system**, per `email-templates.md`). Docs localize via `web/src/pages/DocsPage/loader.ts` — a German mirror under `content/de/**` with per-file English fallback.
- **Adding a language:** add its code to `SUPPORTED_LANGUAGES` and an endonym to `LANGUAGE_ENDONYMS` in `index.ts` (the picker renders from these), then drop `locales/<code>/*.json`. Missing keys fall back to English, so partial coverage degrades gracefully.

## Rules when translating

- **Every new or changed user-facing string goes through `t()`, with the key added to ALL 10 locale files** — not just `en`/`de`. The most common bug this session: an agent changed a string's SHAPE (added `{{count}}`/`{{limit}}` interpolation) in `en`/`de` only, so the other 8 locales kept the stale literal ("3 mind maps" while the real cap was 25). **When you touch a key, touch all 10, and verify parity** — every locale has the same key set and, for interpolated keys, the same `{{placeholders}}`.
- **Plural forms per CLDR:** `en/de/es/fr/it/nl/pt` use `_one`/`_other`; `ja` uses `_other` only; `ru`/`pl` need `_one`/`_few`/`_many`/`_other`. Fill every form the target language requires.
- **Register:** informal second person, consistent per language (de `du`, es `tú`, fr `tu`, pt-BR `você`, it `tu`, nl `je`, pl/ru `ty`/`ты`, ja polite です/ます). Follow `VOICE.md` tone.
- **Protected strings — verbatim in every locale:** brand/product names (Anki, AnkiWeb, Notion, Quizlet, 2anki), format names (PDF/CSV/HTML/Markdown/APKG), OAuth button labels, Stripe plan names + prices, `support@2anki.net`, code/CLI, `{{…}}` tokens, and parser-matched literals (the `front`/`back`/`question`/`answer` CSV header tokens). When unsure whether a token is protected, leave it English and flag it.
- **Business-string policy for "100 cards per month" / "Your monthly limit" (decided 2026-07):** the `/pricing` MARKETING string stays verbatim English (SEO — the `pricingtable` namespace keeps it English in all locales). The IN-PRODUCT limit phrasing (banners, account meta, `LimitPage`, mindmap limit) IS localized — keep the digit, translate the words.
- **Do NOT translate without an explicit decision:** legal docs (`reference/terms`, `reference/privacy` — need legal review) and the SEO landing copy's meta/`<title>`/JSON-LD (needs a routing/canonical strategy; the visible on-page copy is translated, the structured data is not).

## Parallel translation (fan-out)

- One agent per language OR per area, **each owning its own namespace file(s)** — never editing `common.json` or another agent's files, so PRs don't collide. Each agent adds a distinct changelog JSON (`YYYY-MM-DD-<slug>.json`).
- **Before every push, run `pnpm format:check`** (CI's exact tree-wide `oxfmt -c .oxfmtrc.json --check src web/src`), NOT a per-file `oxfmt` — a per-file run passes locally while the tree-wide check fails CI, and a merged format failure poisons every subsequent PR's `static` job. See the format/green-gate gotcha in `CLAUDE.md`.
- **Verify parity before merge:** for a change that alters key shape, grep all 10 locale files for the affected keys and confirm they all match. An agent's "only en/de exist" claim was wrong once — all 10 dirs exist; check, don't trust.
