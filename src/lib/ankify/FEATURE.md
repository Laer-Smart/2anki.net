# ankify — Notion → Anki sync

Background-job factories and pure helpers for the Ankify product (separate from the one-shot conversion path). No HTTP, no SQL, no third-party SDK clients here — those live in `src/services/ankify/`.

## What's here

- `jobs/scheduleAnkifyPolling.ts` — schedules the Notion poll loop (5-min cadence; webhook story is deferred per `Documentation/ankify/notion-webhooks-deferred.md`). Before syncing each subscription from the enabled snapshot, the tick re-checks the row still exists via `findByOwnerAndPageId`; a deck the user deleted between the snapshot and its turn is skipped, so deletion sticks instead of being re-registered by `SyncNotionPageToRacUseCase`'s upsert. The tick forwards `current.notion_object_type` as `knownObjectType` so the use case can skip the doomed `pages.retrieve` on subscriptions already known to point at a Notion database — a database subscription previously re-paid one guaranteed-failing Notion call per tick plus `@notionhq/client` warn spam. `SyncNotionPageToRacUseCase` self-heals the type on the first successful walk (records `page` or `database` via `recordObjectType`); backfill is unnecessary. For `trigger: 'polling'` the use case probes AnkiConnect with a cheap `ping()` (a `version` invoke) before any Notion fetch; if the client is offline it skips the whole tick for that subscription, records a calm `last_error` ("Anki client offline — will retry next tick"), logs one info line, and returns `AnkifyClientOfflineSkip` instead of running the full Notion fetch + sync and aborting at the AnkiConnect step. Manual/webhook triggers do not pre-empt — they run normally and surface `AnkiConnectUnreachableError`.
- `jobs/scheduleAnkifyReaper.ts` — cleans up dead/expired sessions and tokens.
- `scheduler/instance.ts` — singleton scheduler; survives one process lifetime.
- `nextDailyRunAt.ts` — pure cron-ish next-run calculation. Heavily tested.
- `notionWebhookSignature.ts` — HMAC verification for Notion's webhook payloads. Wired into the live `AnkifyWebhookRouter` (mounted at `server.ts`), but no production webhooks reach the route today: Notion-side auto-registration is the deferred piece (see deferred doc), so polling still carries the story. The route is mounted, signature-verified, and access-gated — only unfed.
- `hashCardContent.ts` — `hashCardContent(front, back)` returns a stable SHA-256 hex digest of a card's rendered front and back (null-byte separated, so a boundary shift between the two fields changes the digest). `SyncNotionPageToRacUseCase` stores this on each `ankify_sync_mappings.content_hash` and compares it on every sync; a renderer improvement (e.g. a styling fix) changes the digest even when the Notion page was never edited, so already-synced cards on unedited pages get rewritten via `updateNoteFields` instead of being skipped as "unchanged." A null stored hash (rows that predate the column) is treated as "differs," so the installed base recolours once on its first post-deploy sync.
- `access.ts` — `hasAnkifyAccess(user, subscriptions, autoSyncProductId)` returns true when `users.patreon === true` **OR** `subscriptions` contains an active row whose `stripe_product_id` equals `autoSyncProductId`. Single source of truth for the gate; consumed by `RequireAnkifyAccess` middleware, `ValidateAnkifySessionTokenUseCase`, `AnkifyWebhookRouter`, the mindmap use cases (`Create`/`List`/`Update`), and `ShouldShowAutoSyncPitchUseCase` — the gate now governs Mindmaps and pitch display too, not just Ankify. Pass `process.env.AUTO_SYNC_PRODUCT_ID ?? ''` as the third argument at every call site. Exports `AnkifyAccessUser` and `AnkifyAccessSubscription` interfaces. `AnkifyAccessSubscription.stripe_product_id` is optional so that existing `Subscriptions` DB rows (which lack the column pre-migration) pass the type check — once `pnpm kanel` runs after the migration, the generated type will include it.
- `deckOwnership.ts` — `userOwnsDeck(requestedDeck, subscriptions)` and `ownedDeckNames(subscriptions)`. The cockpit endpoints that accept a deck name from the client (`gui-deck-overview`, `deck-maturity`) must not let an arbitrary deck through — they verify the requested deck is one the caller actually syncs. `ownedDeckNames` derives each subscription's real Anki deck name via `buildDeckName(target_deck, notion_page_title)`; `userOwnsDeck` returns true when the requested deck equals an owned deck name OR is a `::`-hierarchy descendant of one (so database child subdecks resolve). A bare prefix that isn't a hierarchy boundary (`MS3::PharmacologyExtra` vs `MS3::Pharmacology`) is rejected. Pure functions; the use cases call `subscriptions.listByOwner(owner)` and pass the rows in.
- `transforms/limits.ts` — `TRANSFORM_NOTE_CAP_PAID = 250` and `getTransformNoteCap(isPaying)`. v1 caps every paid transform job at 250 notes to dodge the request-timeout problem while durable job tracking is deferred — the controller passes the cap into `TransformApkgUseCase`, which throws `DeckTooLargeError` (carrying `noteCount` + `noteCap`) when the parsed deck is over. The controller maps that error to 413 with a message naming both numbers. The preview endpoint echoes `noteCap` so the page can disable submit and show the over-cap banner before a wasted request. `getTransformNoteCap(true)` returns 250; `getTransformNoteCap(false)` returns 0 — the paywall blocks free users earlier, but the signature leaves the door open to a per-tier cap without rewriting callers. Raise the constant only after durable jobs ship; the 250-note cap is what makes the sync request path safe.
- `transforms/tags.ts` — `parseTagInput(raw)` and `mergeTags(source, extra)`. `parseTagInput` splits the raw string on whitespace runs, strips characters outside `[A-Za-z0-9_:-]`, collapses any colon run of length ≥ 3 down to `::` (Anki's hierarchy separator stays intact), dedupes while preserving first-occurrence order, and caps the result at 10 tags silently. Non-string input returns `[]`. `mergeTags` unions the per-note source tags with the user-supplied extras, source order first, dropping duplicates — never mutates the input arrays. Both the front-end (`TransformPage`) and the controller (`parseTagsField` → `parseTransformBody`) call `parseTagInput` against the same input, so the client and server agree on what the user "really" typed; the use case calls `mergeTags` once before deck build to fold the parsed extras into every output `TransformedNote.tags`. v1 is merge-only — there's no replace mode and no per-card rules. The 10-tag silent cap matches the existing 250-note cap pattern (let users proceed within sensible limits instead of blocking on edge cases). Raise the cap only after we see real adoption. `sanitizeDeckPath(raw)` cleans a user-supplied Anki deck path for the Ankify "choose target deck" feature — it strips control characters, `"`, and `<`, collapses any colon run of length ≥ 3 to `::` (Anki's hierarchy separator), trims each `::`-delimited segment and drops empties, but PRESERVES spaces and commas inside segment names (real deck names carry both). Non-string input returns `''`. `transforms/deckName.ts` exports `buildDeckName(override, title)` — runs the override through `sanitizeDeckPath` and falls back to `Notion Sync::<title>` when the result is empty (title defaults to `Untitled`) — and `buildChildDeckName(parentDeck, childTitle)`, which nests a database child page's title under the parent deck (`<parent>::<child title>`, same `Untitled` fallback, `::` stripped from the title). `SyncNotionPageToRacUseCase` uses the pair to give every database child page its own subdeck (a `target_deck` override stays the parent path); `ResolveConflictUseCase` imports `buildDeckName` so a conflict resolved on a relocated card records the same deck name the sync would, instead of a stale literal. The controller sanitizes the same way before persisting `target_deck`.
- **Source-deck media passthrough.** `parseApkgNotes` (in `src/services/ApkgPreviewService/`) now returns `sourceMedia: SourceMediaFile[]` alongside notes — every `<img src="X.png">` reference in the input deck stays renderable in the output because `TransformApkgUseCase` does two things: (a) merges source bytes with newly-fetched images via `mergeMedia(sourceMedia, fetchedMedia)` (fetched wins on filename collision; with our `2anki-<hash>` naming the collision case is functionally impossible), and (b) walks each transformed note's fields with `transforms/attachReferencedMedia.ts` to scan for `<img src="X">` references whose filename is in the merged-media set, and merges those filenames into `note.media`. **Both steps are required.** `create_deck.py` packages files into the `.apkg` only when they appear on some card's `media` array (`media_files = media_files + card["media"]`) — writing bytes to disk via `exporter.addMedia` is necessary but not sufficient; without the per-note reference, the genanki step drops the file. Do not regress this by dropping `sourceMedia` from the parse result, the merge step in the use case, the `attachReferencedMedia` step, or the exporter's `addMedia` calls. The source media filename is reduced to its `path.basename` at ingestion in `collectSourceMedia`, and `CustomExporter.addMedia` resolves the destination through `resolveSafeEntryPath` against the workspace before writing — a manifest entry can never write outside the deck workspace (CWE-22). Keep both guards: dropping the basename or the resolve-and-assert reopens the path-traversal write.
- `transforms/` — pure helpers for the bulk `.apkg`-input transform pipeline. `types.ts` defines `ParsedNote` and `TransformedNote` as N-field shapes — `fields: string[]` carries every field value in ordinal order, `fieldNames: string[]` carries the matching field names from the note type, and `modelName` preserves the original note-type name so it can round-trip through the emit side. `frontFieldIndex` / `backFieldIndex` carry the field indices the source note type's first template actually renders as question and answer (derived by `resolveTemplateFieldIndices` in `src/services/ApkgPreviewService/` from the model's qfmt/afmt; fallback 0/1 when templates are absent or reference nothing recognizable). The `getFrontField` / `getBackField` helpers read through those indices — never assume front=field 0, back=field 1 by position; third-party vocab exports routinely render Back from the third field, and ordinal assumptions put transform output in the wrong column (#3119). `FieldSelection { sourceField?, targetField? }` parameterizes which field a transform reads from and writes to (defaults: source=template front, target=template back); an explicit selection from the client always wins. `TransformName` catalog (`translate_back`, `add_example`, `cloze_front`, `add_hint`, `add_image`), the supported target-language list for `translate_back`, and the `ImageSource` enum (`pexels`, `wikimedia`) for `add_image` round out the type surface. `prompts.ts` builds the per-note system + user prompt for each Claude-driven transform; `add_image` does NOT use a Claude prompt — it fetches a real photo or diagram via the image-source API and appends an `<img>` tag to the target field. `applyTransformedFields.ts` maps the LLM JSON result back onto `fields[targetField]`, flipping `modelKind` to `'cloze'` for `cloze_front`. The `TransformedNote.media` optional field carries per-note image filenames so the .apkg packager can include them in the deck's media folder (set by `add_image`, unused by the LLM-driven transforms). No Claude calls here — those live in `src/services/ankify/transformService.ts`; image fetches live in `src/services/ankify/imageFetchService.ts`. Plan gating for the transform surface uses `isPaying` (paid plan), not `hasAnkifyAccess` — Ankify and Transform are distinct paid surfaces.

## Things to know before editing

- **Gate:** Ankify is accessible to users with `patreon = true` (lifetime/grandfathered) **or** an active `subscriptions` row with `stripe_product_id === AUTO_SYNC_PRODUCT_ID` ($30/mo Auto Sync). Use `hasAnkifyAccess` from `access.ts`; do not reintroduce hard-coded emails. To widen further, edit `access.ts` only. Every call site must pass the subscription rows and `process.env.AUTO_SYNC_PRODUCT_ID ?? ''` — the caller is responsible for fetching subscriptions from the DB (see `RequireAnkifyAccess` for the pattern).
- **`NOTION_WEBHOOK_SECRET` is read by the live router today** — `AnkifyWebhookRouter` reads a single global `process.env.NOTION_WEBHOOK_SECRET` and 500s without it. This is the interim single-secret shape; the deferred work replaces it with per-subscription secrets + auto-registration via the user's OAuth token. Don't build new setup around the global var expecting it to last — it's a stopgap, not the target shape.
- **Pure functions only.** If you reach for `axios`, `knex`, or `@notionhq/client`, the code belongs in `services/ankify/`.
- The poll cadence is intentional — Notion's free-tier rate limits punish anything tighter. Don't drop it without measuring.
