# ankify — Notion → Anki sync

Background-job factories and pure helpers for the Ankify product (separate from the one-shot conversion path). No HTTP, no SQL, no third-party SDK clients here — those live in `src/services/ankify/`.

## What's here

- `jobs/scheduleAnkifyPolling.ts` — schedules the Notion poll loop (5-min cadence; webhook story is deferred per `Documentation/ankify/notion-webhooks-deferred.md`). Before syncing each subscription from the enabled snapshot, the tick re-checks the row still exists via `findByOwnerAndPageId`; a deck the user deleted between the snapshot and its turn is skipped, so deletion sticks instead of being re-registered by `SyncNotionPageToRacUseCase`'s upsert. The tick forwards `current.notion_object_type` as `knownObjectType` so the use case can skip the doomed `pages.retrieve` on subscriptions already known to point at a Notion database — a database subscription previously re-paid one guaranteed-failing Notion call per tick plus `@notionhq/client` warn spam. `SyncNotionPageToRacUseCase` self-heals the type on the first successful walk (records `page` or `database` via `recordObjectType`); backfill is unnecessary. For `trigger: 'polling'` the use case probes AnkiConnect with a cheap `ping()` (a `version` invoke) before any Notion fetch; if the client is offline it skips the whole tick for that subscription, records a calm `last_error` ("Anki client offline — will retry next tick"), logs one info line, and returns `AnkifyClientOfflineSkip` instead of running the full Notion fetch + sync and aborting at the AnkiConnect step. Manual/webhook triggers do not pre-empt — they run normally and surface `AnkiConnectUnreachableError`. The same calm-skip covers the "no active Ankify client row at all" case: on `trigger: 'polling'`, an owner who linked Notion + a subscription but never provisioned a running client takes the identical `skipOfflinePoll` path (calm `last_error`, one info line, `AnkifyClientOfflineSkip`) instead of throwing `NoActiveAnkifyClientError` and logging a full stack every 5-min tick. Manual/webhook triggers still surface `NoActiveAnkifyClientError` normally. Reusing the "Anki client offline" message keeps the cockpit's `isCalmOfflineError` classifier rendering these as the muted offline line, not a danger error.
- `jobs/scheduleAnkifyReaper.ts` — cleans up dead/expired sessions and tokens.
- `scheduler/instance.ts` — singleton scheduler; survives one process lifetime.
- `nextDailyRunAt.ts` — pure cron-ish next-run calculation. Heavily tested.
- `notionWebhookSignature.ts` — HMAC verification for Notion's webhook payloads. Wired into the live `AnkifyWebhookRouter` (mounted at `server.ts`), but no production webhooks reach the route today: Notion-side auto-registration is the deferred piece (see deferred doc), so polling still carries the story. The route is mounted, signature-verified, and access-gated — only unfed.
- `hashCardContent.ts` — `hashCardContent(front, back)` returns a stable SHA-256 hex digest of a card's rendered front and back (null-byte separated, so a boundary shift between the two fields changes the digest). `SyncNotionPageToRacUseCase` stores this on each `ankify_sync_mappings.content_hash` and compares it on every sync; a renderer improvement (e.g. a styling fix) changes the digest even when the Notion page was never edited, so already-synced cards on unedited pages get rewritten via `updateNoteFields` instead of being skipped as "unchanged." A null stored hash (rows that predate the column) is treated as "differs," so the installed base recolours once on its first post-deploy sync.
- `access.ts` — `hasAnkifyAccess(user, subscriptions, autoSyncProductId)` returns true when `users.patreon === true` **OR** `users.ankify_access === true` **OR** `subscriptions` contains an active row whose `stripe_product_id` equals `autoSyncProductId`. The `ankify_access` boolean is a first-class comp grant: it turns on Auto Sync (and everything else the gate governs) for a single user _without_ making them a lifetime/`patreon` account and _without_ a synthetic `subscriptions` row (which the Stripe reconcile would either purge or double-count). Set it with `UPDATE users SET ankify_access = true WHERE id = <id>`; revoke by flipping it back to false. Single source of truth for the gate; consumed by `RequireAnkifyAccess` middleware, `ValidateAnkifySessionTokenUseCase`, `AnkifyWebhookRouter`, the mindmap use cases (`Create`/`List`/`Update`), and `ShouldShowAutoSyncPitchUseCase` — the gate now governs Mindmaps and pitch display too, not just Ankify. Pass `process.env.AUTO_SYNC_PRODUCT_ID ?? ''` as the third argument at every call site. Exports `AnkifyAccessUser` and `AnkifyAccessSubscription` interfaces. `AnkifyAccessSubscription.stripe_product_id` is optional so that existing `Subscriptions` DB rows (which lack the column pre-migration) pass the type check — once `pnpm kanel` runs after the migration, the generated type will include it.
- `deckOwnership.ts` — `userOwnsDeck(requestedDeck, subscriptions)` and `ownedDeckNames(subscriptions)`. The cockpit endpoints that accept a deck name from the client (`gui-deck-overview`, `deck-maturity`) must not let an arbitrary deck through — they verify the requested deck is one the caller actually syncs. `ownedDeckNames` derives each subscription's real Anki deck name via `buildDeckName(target_deck, notion_page_title)`; `userOwnsDeck` returns true when the requested deck equals an owned deck name OR is a `::`-hierarchy descendant of one (so database child subdecks resolve). A bare prefix that isn't a hierarchy boundary (`MS3::PharmacologyExtra` vs `MS3::Pharmacology`) is rejected. Pure functions; the use cases call `subscriptions.listByOwner(owner)` and pass the rows in.
- `transforms/tags.ts` — `sanitizeDeckPath(raw)` cleans a user-supplied Anki deck path for the Ankify "choose target deck" feature — it strips control characters, `"`, and `<`, collapses any colon run of length ≥ 3 to `::` (Anki's hierarchy separator), trims each `::`-delimited segment and drops empties, but PRESERVES spaces and commas inside segment names (real deck names carry both). Non-string input returns `''`. `transforms/deckName.ts` exports `buildDeckName(override, title)` — runs the override through `sanitizeDeckPath` and falls back to `Notion Sync::<title>` when the result is empty (title defaults to `Untitled`) — and `buildChildDeckName(parentDeck, childTitle)`, which nests a database child page's title under the parent deck (`<parent>::<child title>`, same `Untitled` fallback, `::` stripped from the title). `SyncNotionPageToRacUseCase` uses the pair to give every database child page its own subdeck (a `target_deck` override stays the parent path); `ResolveConflictUseCase` imports `buildDeckName` so a conflict resolved on a relocated card records the same deck name the sync would, instead of a stale literal. The controller sanitizes the same way before persisting `target_deck`.
- `transforms/types.ts` — `ParsedNote` and `SourceModelKind` are the N-field parse shape shared by `ApkgPreviewService.parseApkgNotes` and the CSV export path (`buildCsvFromApkgNotes`). `fields: string[]` carries every field value in ordinal order, `fieldNames: string[]` the matching field names, `modelName` the original note-type name, and `frontFieldIndex` / `backFieldIndex` the indices the source note type's first template renders as question and answer (derived by `resolveTemplateFieldIndices` in `src/services/ApkgPreviewService/` from the model's qfmt/afmt; fallback 0/1 when templates are absent).

## Things to know before editing

- **Gate:** Ankify is accessible to users with `patreon = true` (lifetime/grandfathered), `ankify_access = true` (a per-user comp grant — see `access.ts`), **or** an active `subscriptions` row with `stripe_product_id === AUTO_SYNC_PRODUCT_ID` ($30/mo Auto Sync). Use `hasAnkifyAccess` from `access.ts`; do not reintroduce hard-coded emails. To comp a single user (e.g. a trial), set `users.ankify_access = true` — never `patreon = true` (that grants full lifetime) and never a hand-written `subscriptions` row (fake payment data). To widen further, edit `access.ts` only. Every call site must pass the subscription rows and `process.env.AUTO_SYNC_PRODUCT_ID ?? ''` — the caller is responsible for fetching subscriptions from the DB (see `RequireAnkifyAccess` for the pattern).
- **`NOTION_WEBHOOK_SECRET` is read by the live router today** — `AnkifyWebhookRouter` reads a single global `process.env.NOTION_WEBHOOK_SECRET` and 500s without it. This is the interim single-secret shape; the deferred work replaces it with per-subscription secrets + auto-registration via the user's OAuth token. Don't build new setup around the global var expecting it to last — it's a stopgap, not the target shape.
- **Pure functions only.** If you reach for `axios`, `knex`, or `@notionhq/client`, the code belongs in `services/ankify/`.
- The poll cadence is intentional — Notion's free-tier rate limits punish anything tighter. Don't drop it without measuring.
- **AnkiConnect is strictly per-user — do NOT add a subscription-ownership gate on review/grade.** The client is resolved via `clients.findActiveByOwner(owner)` (owner from `RequireAnkifyAccess` → `res.locals.owner`) against that one client's `anki_port` + `anki_connect_api_key`, so a user can only ever reach their own container/collection. A `userOwnsDeck(deck, subscriptions)` gate on the review/grade path adds ZERO cross-user security and breaks whenever the synced deck's real Anki name ≠ `buildDeckName(target_deck, notion_page_title)` — it was removed in #3358 after surfacing as a false "Anki isn't connected" on a lifetime user's own deck. The real boundary is the per-user client + `escapeDeckQueryValue` + fail-closed ease 1–4 validation + a `cid:` existence probe before `answerCards`. Keep `userOwnsDeck` only on surfaces that echo deck names back cosmetically (`gui-deck-overview`, `deck-maturity`). The frontend must surface offline-vs-error, never collapse every non-200 to `{connected:false}`. Note: the `/ankify` WorkspaceBar "Anki running" status = `containerReady && ankiWebLinked`, NOT an AnkiConnect ping — it can read "running" while stats/review correctly report unreachable.
- **Reviewer audio: resolve `[anki:play:*]`, never strip it.** Language decks (jlab, subs2srs) keep audio in a note FIELD (e.g. `Audio = [sound:…mp3]`) and the card template renders Anki's `[anki:play:q:0]` / `[anki:play:a:N]` directive — so AnkiConnect `cardsInfo` `question`/`answer` carries `[anki:play:…]` with no inline `[sound:]`. `GetReviewCardUseCase` (#3371) fetches `notesInfo([note])`, collects the note's field `[sound:FILE]` filenames in field order → `noteSounds[]`, and `inlineReviewMedia` resolves `[anki:play:q|a:N]` → `<audio src="data:…">`. Stripping the directive silences these decks. Cards with no `[sound:]` and no `[anki:play]` genuinely have no AnkiConnect-renderable audio. Diagnose a "no sound" report by querying the user's container AnkiConnect (`owner → ankify_clients.anki_port` + api key) with `findNotes` / `notesInfo` / `cardsInfo`.
