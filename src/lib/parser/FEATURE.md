# parser — Notion → Anki conversion

The hot path. Every Notion page, HTML export, markdown file, or zip the user uploads ends up running through this directory. Performance and correctness here directly move the 300K-user goal.

## Entry points

- `DeckParser.ts` — orchestrates a full conversion: file → blocks → notes → packaged `Deck`. `DeckParserInput` accepts an optional `pdfCredential?: string` that is threaded through to the PDF extraction layer; never log or store it. It also accepts an optional `userId?: number | null` that `PrepareDeck` threads into the Claude path so the AI converter's `ai_conversion_completed` telemetry can attribute cost back to the user; `null` is the safe default for anonymous or worker contexts where owner isn't known. Exposes `writeDeckInfo(ws)` alongside `build(ws)` — both async (the card-processing phase fetches remote image bytes): the former runs the full card-processing phase and writes `deck_info.json` but stops before spawning Python, resolving to the path — used by the batch build path in `getPackagesFromZip`.
- `Deck.ts` / `Note.ts` / `Package.ts` — domain types for what a card looks like. `Note` carries `customFields?`, `customFieldNames?`, and `customModelName?` for the N-field path used by the apkg-input transform pipeline — when present, the Python card-generator builds a genanki Model with the named fields instead of the default 2-field Basic/Cloze shape, preserving the source deck's note-type structure. `Note` also carries `hierarchy?` plus `h1?/h2?/h3?` for the Notion heading-fields path (`CardOption.template === 'hierarchy'`) — when `hierarchy` is true the Python side emits the `n2a-hierarchy` model (H1/H2/H3/Question/Answer fields) instead of basic; `reversed()` copies the heading fields onto the reversed note.
- **Per-card reverse (`🔄`).** A `🔄` (or its HTML-entity form `&#x1F504;`) at the start of a card front marks that single card for a reversed companion note, independent of the deck-wide `reversed`/`basic-reversed` settings. `Note.hasRefreshIcon()` detects it (on `name` only); `Note.reversed(input)` builds the swapped companion with marker stripped from both faces; `Note.stripRefreshIcon()` removes the marker (emoji or entity, plus surrounding whitespace) from the source note's faces. The pure stripper is `removeRefreshIcon(value)`. Both conversion paths honor it with the same OR-logic (`basicReversed || hasRefreshIcon()`): `BlockHandler.handleListBlock` (Notion API) and `DeckParser.processPayload` (HTML/markdown upload). Detect first, create the reversed note, then strip — stripping before detection would break the trigger. The marker must never reach a rendered card face. Keep the two-plain-notes behavior; switching to Anki's native "Basic (and reversed card)" model is a deferred product decision. **Reversal is basic-only on both paths.** The Notion path guards with `isBasicType` (set false for cloze/mcq/input cards); `DeckParser.processPayload` mirrors it with `Note.isReversibleBasic()` (false for mcq, a real input card, or a note whose rendered face carries a `{{c` cloze deletion). Reversing a cloze card emits a basic companion that shows raw `{{c1::…}}` markup, and deck-wide `reversed` on an mcq note swaps its empty back into `name` so `CleanCards` silently drops it — both gated off by the predicate (#3244). Do not reverse or swap a note that is not `isReversibleBasic()`.
- `countEmptyBacks.ts` — pure `countEmptyBacks(cards, getBack, getFront)`: counts cards whose back is empty/whitespace while the front is non-empty (an empty-front card is a different discard and is excluded). Used by `DeckParser.processPayload` (which exposes `emptyBackCount` + `parsePathSignature()`) and `BlockHandler` (which exposes `emptyBackCount` + `cardCount`) to measure silent content loss **before** `Deck.CleanCards` drops the empties — the nested-toggle "empty backs" incident produced garbage with no error or log. These feed the ops `conversion_output` / `parse_path_signatures` drift metrics; do not move the count after `CleanCards` or it reads zero.
- `findNotionToggleLists.ts` — toggle-list detection on the Notion HTML export. The original primitive the project was built on.
- `guessMarkdownCards.ts` — fallback for raw markdown without toggle structure.
- `exporters/CustomExporter.ts` + `embedFile.ts` — write the `.apkg` (sqlite + media) for download. `configure()` wraps `JSON.stringify` to convert `RangeError` (V8 "Invalid string length") into `DeckTooLargeError` so the caller gets a typed error instead of a raw engine exception. When `settings.ttsAutoDetect` is true it samples the first 25 card fronts via `lib/anki/detectCardLanguage`, sets `settings.frontLang` (`'ja' | 'zh' | 'ko' | 'en'`), and emits one `tts_lang_injected` analytics event. A manual pick (`settings.ttsManualLang` + `settings.ttsManualSide` = `front | back | both`) takes precedence — when a manual lang is set, `applyAutoDetectedLang` short-circuits so `frontLang` stays empty. `create_deck/helpers/get_model.py` resolves the effective front/back lang via `resolve_tts_langs` (manual wins over `frontLang`) and prepends `{{tts <lang>:Front}}`/`{{tts <lang>:Back}}` (or `:Text`/`:Extra` for cloze) to the matching templates of basic/cloze/input cards; MCQ keeps its dedicated `mcqTts*` keys. `deckInfoPath()` returns the absolute path to the written `deck_info.json` — used by the batch build path.
- `WorkSpace.ts` — `Workspace.subdir(parentLocation)` creates an isolated subdirectory workspace within an existing workspace location. Used by the batch build path to give each deck in a batch its own `deck_info.json` without collision.
- `exporters/DeckTooLargeError.ts` — typed error thrown when the deck payload is too large to serialize. Caught by `UploadService` and mapped to a clean 400 response.
- `extractPdfText.ts` — extracts per-page text from a PDF buffer using `pdf-parse`. Accepts an optional `credential` parameter passed as `{ userPassword }` to `pdf-parse`. A custom `pagerender` runs both `getTextContent` (preserving the default text-accumulation behavior) and `getOperatorList`, so each `PdfPage` carries `imagePaintCount` — the number of raster-image paint ops on that page (resolved from the pinned `v1.10.100` pdf.js `OPS`; image masks are excluded). The render also emits a `\f` per page so `splitIntoPages` gets clean page boundaries. Returns `isDrmLocked: true` when average chars/page is below 10 (DRM or image-only PDF); returns `needsCredential: true` when `pdf-parse` throws a `PasswordException` (encrypted PDF with no or wrong credential). Logged via `[extractPdfText] result` (with `pagesWithImage`) for production monitoring. On load it neutralizes two browser-global accesses pdf.js makes in Node (both harmless to extraction but log-flooding): it sets `disableFontFace = true` on the pdf.js **global settings object** (`pdfjs.PDFJS`, the object `getDefaultSetting` reads — NOT the module root, which is a different object; setting the flag on the root is a silent no-op and leaves `ReferenceError: document is not defined` flooding the log), and it installs a no-op `Image` global shim so `loadJpegStream`'s `new Image()` decode path resolves quietly instead of throwing `ReferenceError: Image is not defined` (pdf-parse calls `getDocument(buffer)` with no params, so `nativeImageDecoderSupport` can't be set to skip that path).
- `synthesizeCardsFromPdf.ts` — converts `PdfPage[]` into `PdfCard[]` using the slide-pair model: page N is the card front, page N+1 is the back. Blank pages in either slot are skipped. Pure function, no I/O. This is the manual-flag (`pdf-extract-text`) path's synthesizer — its output contract is frozen for users who chose the flag; behavior changes go into `synthesizeCardsFromPdfHeadings.ts` instead.
- `synthesizeCardsFromPdfHeadings.ts` — converts `PdfPage[]` into `PdfCard[]` by splitting on heading-like lines: a trimmed non-blank line under 60 chars whose next non-blank line is longer starts a new card (front = heading, back = following lines until the next heading). A candidate is **disqualified** when it ends in sentence/clause punctuation (`.` `,` `;` — `?` and `:` stay valid headings), starts with a list marker (`•`/`-`/`*`/`1.`/`1)` etc.), starts lowercase, or continues a wrapped line (previous line ≥ 60 chars without terminal punctuation) — pdf.js emits `\n` at every visual line wrap, so without these filters the short tail of a wrapped bullet becomes a junk card front and a bulleted lecture PDF explodes into hundreds of cards. Headings with no body and text before the first heading are dropped. Used only by the auto-detected PDF text path (`convertPdfTextToHtmlAuto`), never by the manual flag path. The auto path also flags `overSplit` when output exceeds `MAX_CARDS_PER_PAGE` (15) cards/page — `PrepareDeck` then routes to the page-image fallback instead of shipping a card explosion, and logs `cardsPerPage` on both branches. Pure function, no I/O.
- `isTextShapedPdf.ts` — pure heuristic over a `PdfExtractionResult`: true when ≥80% of pages have extractable text AND average ≥200 chars/page AND the PDF is neither DRM-locked nor awaiting a password — UNLESS the PDF looks like a scanned/exported card deck, which always routes to the image-pair path. A card deck is detected when a raster image covers ≥90% of pages (`imagePaintCount > 0`) AND the per-page text length is uniform (coefficient of variation ≤ 0.5). That conjunction separates scanned cards (one full-page image per side, a thin/uniform OCR or export text layer that can still clear the 200-char bar — #3402) from genuine prose (no full-page images) and from slide decks (figures on only some pages, varied text length). Gates the automatic PDF text path in `PrepareDeck`; everything ambiguous stays on the image-pair path. Thresholds exported as constants — tune there, not at call sites.
- `xlsx/` — spreadsheet → cards path.
- `parsers/parseOpml.ts` — pure function, input OPML string → `MindmapData`. Uses cheerio in xmlMode to walk `<outline>` elements. Each parent→child outline pair becomes one edge. Falls back to `_note` attribute when `text` is absent. Throws on malformed XML or missing `<body>`.
- `parsers/parseBrainstormsJson.ts` — pure function, input Brainstorms JSON string → `MindmapData`. Validates `nodes[]` and `edges[]` arrays; throws if either is absent or if the text is not valid JSON. Accepts the shape `{ nodes: [{id, label}], edges: [{source, target}] }`.
- `parsers/parseAnkiAppXml.ts` — pure function, input AnkiApp deck XML string → `{ name, notes: Note[], skippedMediaOnlyCount }`. Cheerio in xmlMode. Field `sides` attributes (`"11"` both, `"10"` front, `"01"` back) drive front/back placement; falls back to positional (first field = front, rest = back) when definitions are missing or mismatched — AnkiApp's own docs ship a mismatched example. `rich-text` goes through `sanitizeCardHtml`, `markdown` through `markdownToHTML`; `tts`/`tex`/`code`/`japanese`/`chinese`/`translation` degrade to escaped plain text; `audio`/`img`/`video` blob refs are dropped, and a card left with an empty side is skipped and counted in `skippedMediaOnlyCount`. Throws `AnkiAppExportError` with user-facing copy (`ANKI_APP_MALFORMED_XML_MESSAGE`, `ANKI_APP_NO_CARDS_MESSAGE`) — the worker passes these messages straight to the client, so keep them user-readable.
- `canary/scheduleParserCanary.ts` — daily job (03:00 UTC) that runs the fixture corpus through the live parser and emails `SUPPORT_EMAIL_ADDRESS` on any count divergence. Wired in `server.ts`.
- `sourceUnits/` — deterministic pre-pass that normalises raw PPTX and free-form notes into a structured intermediate before any AI step. Exports `SourceUnit` (id, visibleText, speakerNotes, role), `extractPptxSourceUnits(buffer)` (async, uses `fflate` to unzip and regex to parse slide/notes XML), and `extractNotesSourceUnits(content, format)` (pure, splits markdown or HTML by heading sections). No HTTP, no SDK, no DB. The AI generation step reads this intermediate and cites `SourceUnit.id` per card to make coverage auditable.

## Flow

1. Upload lands in `controllers/Upload/` and dispatches to `usecases/uploads/`.
2. `DeckParser` reads the file, picks a strategy (toggle list / nested bullets / markdown / xlsx), normalises styles via `extractStyles`, and emits `Note[]`.
3. `CustomExporter` writes the apkg using `better-sqlite3` and the media bundle.
4. The download URL goes back to the user via `JobService` / `DownloadService`.

## Constraints

- **Memory:** users upload 1 GB+ Notion exports. Stream where possible; never `readFileSync` an upload. The express body limit is `1000mb` for a reason.
- **CPU:** parsing happens inline on the request worker. Anything > 50 ms per card needs a justification; anything > 5 s of total work belongs in a background job.
- **Determinism:** the same input must produce the same `.apkg`, including the note GUID. Tests rely on this.
- **No comments inside cards.** The conversion preserves user content as-is — don't inject "generated by" footers.

## MCQ detection

**Opt-in.** The MCQ pipeline is disabled by default. Detection only fires when the user has enabled it under Card options (`mcq-enabled` key on `CardOption`, default `false`). When off, both the Notion-HTML and markdown paths skip MCQ classification entirely — output is bit-for-bit identical to pre-MCQ behaviour.

Two entry points when enabled: `DeckParser.extractCards()` classifies a Notion toggle as MCQ when its children are either `to_do` blocks (one with `checked: true`) or bulleted list items (one fully bolded with `<strong>`). For markdown uploads, `handleNestedBulletPointsInMarkdown.ts::buildNoteFromBack` runs `detectMarkdownMCQ` against the markdown-it-rendered HTML of the back side, matching `<ul><li><input type="checkbox" checked=""></li>...</ul>` (GFM task list via `markdown-it-task-lists`). When a markdown note is classified as MCQ, `note.back` is cleared — the options live in `note.options` and the Python builder writes them into the dedicated `Multiple Choice` field; leaving the back populated would cause the options to render twice (once in the choices list, once under "Explanation").

**`isMCQ` predicate** (Notion HTML) lives in `findNotionToggleLists.ts` and is called from `DeckParser.extractCards()`. It inspects `<span class="checkbox-on">` for the to-do path and `<strong>` annotations for the bold-fallback path. It returns the index of the marked option, or `-1` if none or more than one is marked.

**`detectMarkdownMCQ` predicate** (markdown) lives in `findNotionToggleLists.ts`. It walks the back-side DOM, requires every `<ul>` it sees to consist entirely of `<li>` elements whose first child is an `<input type="checkbox">`, and returns `{ isMcqShape, correctIndex, options }`. The exact-one-checked rule mirrors the Notion path.

**`detectNotionApiMCQ` predicate** (Notion sync at `/notion`) lives in `findNotionToggleLists.ts`. The Notion SDK renderer emits `<ul class="to-do-list"><li><div class="checkbox checkbox-on|checkbox-off">…</li></ul>` for to-do children; this detector looks for that shape and returns the same result type as the others. Called from `services/NotionService/BlockHandler` when `mcqEnabled` is true.

**`Note` fields for MCQ cards:**

- `mcq: boolean` — true when the note was classified as MCQ
- `options: string[]` — the text of each option in order
- `correctIndices: number[]` — index of the marked option (v1 always has exactly one)
- `isValidMCQNote()` — returns true when `mcq`, at least 2 options, and exactly one `correctIndices` entry

**Failure modes (fall back to Basic/Cloze, counted in `mcqSkippedCount`):**

- Zero markers found in an otherwise MCQ-shaped toggle
- Two or more markers (ambiguous)
- Only one option

**Count threading:** `DeckParser.extractCards()` returns `{ cards, mcqCount, mcqSkippedCount }`. These are stored on the `Deck` object and summed in `PrepareDeck`, then sent as `X-MCQ-Count` / `X-MCQ-Skipped-Count` response headers.

**Dropped-image threading:** `DeckParser.droppedImageCount` increments once per `<img>` that was meant to be bundled but couldn't be: a remote fetch returning null in `embedRemoteImage`, a local/zip path that `embedFile` couldn't resolve in `embedLocalOrZipImage` (toggle/HTML path), or the same resolve miss in `embedImagesInHtml` (markdown heuristic path). A successfully embedded or externally-linked image never reaches a drop; a genuine third-party URL left untouched in the markdown path is not counted (no remote fallback there yet). `PrepareDeck`/`prepareDeckInfoOnly` carry it out as `droppedImageCount`, `getPackagesFromZip` and the upload worker put it on each `Package`, and `UploadService` sums it across packages — `X-Dropped-Assets` on a single-deck sync response, `droppedImageCount` on the batch JSON. The Upload page renders the shared `ImageDropNotice` (source `upload`) from that count, mirroring the Notion-path notice. Claude-branch conversions never run these embed paths, so their count is 0.

**Authoring guide:** `web/src/pages/DocsPage/content/cards/mcq.md` — user-facing guide rendered at `/documentation/cards/mcq`. Update both the body and the brief MCQ section in `card-types.md` when the detection contract changes.

## Block-decision classifier (`intent/`)

`intent/classifyBlock.ts` exports a pure function `classifyBlock(input, rules)` that mirrors the block-filtering predicate inside `BlockHandler.findFlashcardsFromPage`. It returns one of three decisions:

- `'card'` — the block type is in `flashcardTypes`, or it is a toggleable heading and `'toggle'` is in `flashcardTypes`.
- `'recurse'` — `type === 'child_page'` (BlockHandler will recurse into it).
- `'skip'` — everything else.

This is the single source of truth for "what will the parser do with this block?" Both `BlockHandler` and the preview endpoint use it — keeping them in sync by construction. `ClassifyInput` carries `type` and `hasToggleableHeading`; `ClassifyRules` carries `flashcardTypes` from `ParserRules.flaschardTypeNames()`. Pure function — no I/O, no SDK imports.

## Things to know before editing

- `DeckParser.test.ts` is the integration safety net. Anything that changes the output shape needs a green run there.
- `__fixtures__/notion-html-2024/` is the permanent regression corpus for the late-2024 Notion HTML export format. Add new Notion export patterns here rather than as ad-hoc inline strings in tests.
- **Notion 2026 `details.toggle` format.** Notion's newest export emits toggles as a bare `<details class="toggle">` (with a `<div class="indented">` body) directly under `.page-body` — no `<ul class="toggle"><li>` wrapper and no `display:contents` divs. `DeckParser.reshapeBareToggleDetails` (called first in `loadAndNormalizeDOM`, before `hasNotionNewExportFormat` is read) rewrites each `details.toggle` into the exact shape the 2024 `display:contents` export already produces: it drops the `toggle` class off the `details`, converts the `<div class="indented">` body to `<div style="display:contents">`, and wraps the `details` in `<div style="display:contents"><ul class="toggle"><li>…</li></ul></div>` **in place** (`cheerio.wrap`, which never moves children). After the reshape, `hasNotionNewExportFormat` returns true and the 2026 page flows through the identical normalization + nested-toggle cleanup path as the 2024 format, so their card output is byte-identical (modulo the Notion block `id`/`dir` attributes). This is what fixes nested/cascading toggles: the old `wrapBareToggleDetails` left the page as `display:contents`-free, so it fell to `removeNestedTogglesLegacy`, which strips plain-text summaries down to headings only — an empty card back for facts stored in nested summaries (issue #3493, the follow-up nested case #3494 missed). The reshape selector is specific to this format: 2024/legacy details carry no `toggle` class (the class sits on the wrapping `ul`), so those are untouched. Fixtures: `__fixtures__/notion-details-toggle-2026.html` (flat) and `__fixtures__/notion-nested-toggle-2026-capitals.html` (3-level nested, facts in leaf summaries).
- **When a user reports "conversion suddenly broke, I changed nothing", suspect Notion export-format drift first.** External-format drift (the `details.toggle` change above is one instance) is invisible to code review and unit tests until a real export is diffed — like signed-URL expiry, the review bot won't model it. Get a fresh export from the reporter and diff its markup against the parser's selectors (`findNotionToggleLists`, `hasNotionNewExportFormat`, the `.page-body >` selectors) before touching extractor logic.
- `helpers/handleClozeDeletions.ts` merges adjacent `<code>` siblings before numbering clozes. This handles the case where Notion emits one `<code>` per formatting run (bold, italic, color) within a single cloze span.
- **Group cloze blanks per toggle.** Opt-in via the `group-cloze-per-toggle` option (`CardOption.groupClozePerToggle`, **default off**), passed as the second arg to `handleClozeDeletions(input, groupClozePerToggle)` from both `DeckParser.transformCard` call sites. When on, every auto-numbered blank inside one toggle collapses onto the seed cloze number from `findHighestClozeNumber` instead of incrementing — so N `::` markers in a single toggle land on one card that reveals together. Explicit user-typed `cN::` clozes round-trip untouched; only autos collapse. The `next()` closure is the one sanctioned branch inside `handleClozeDeletions` — it gates the post-increment, nothing else. Off (the default) is byte-for-byte the prior behavior. v1 covers the inline-code/toggle path only; the code-block path (`getClozeDeletionCard.ts`) is out of scope.
- **Cloze markers in toggle content (not just the header).** Opt-in via the `cloze-from-toggle-content` option (`CardOption.clozeFromToggleContent`, **default off**); it requires `isCloze` to also be on. When both are on, `DeckParser.transformCard` and the Notion-sync `BlockHandler` both check the toggle: if the header (front) has no inline cloze code but the content (back) does, the card becomes a cloze whose `Text` is the content with `<code>` converted to `{{cN::…}}` and whose `Extra` is the header (the header becomes the hint). With the option off, a toggle whose body has inline code stays a normal Basic card (header front, body back). The header-cloze case is unchanged — header cloze still produces a cloze whose `Text` is the header. `helpers/hasInlineClozeCode.ts` is the shared trigger: it returns true only for a `<code>` element that is **not** inside a `<pre>`, so fenced code blocks in a toggle stay Basic and rich content (tables, lists) in the back survives into the cloze `Text`. Both paths route through `handleClozeDeletions` for the actual conversion so the client and server agree on numbering.
- `helpers/handleOverlappingCloze.ts` powers the `overlapping-cloze` option (`off` | `show-all` | `windowed`, gated behind `cloze`). Pure: `(items, style) => string[]` returns N note bodies, body _i_ wrapping item _i_ in `{{c1::…}}` and the rest rendered per style — `show-all` keeps the whole list visible as context, `windowed` keeps only one line before and one after. **Exactly one `c1` cloze per body, never `c1..cN` in one body** — Anki buries siblings, so one cloze per note is the whole point; the colocated test asserts this. There are four entry points, all fanning items out into N cloze `Note`s (each `cloze = true`, body in `name`, empty `back`): (1) `DeckParser.processPayload` via `buildOverlappingClozeNotes(card)` when a note's `back` holds a `<ul>`/`<ol>` of ≥2 `<li>` items (the Notion toggle-with-list shape), mirroring the `basicReversed` fan-out; (2) `DeckParser.handleHTML` via `buildPageListOverlappingNotes(dom)` for a **top-level page list** (not inside a toggle), which `extractCardsFromLists` would otherwise flatten into one basic card per `<li>` with empty `back` — so the `processPayload` hook never sees the items. The page-list path reads the `.page-body` HTML and replaces the per-item flattening only when the page reduces to a list of ≥2 items. (3) `DeckParser.handleHTML` via `buildPageParagraphOverlappingNotes(dom)` for a **page whose sole meaningful content is one prose block** (a single `<p>` or `<blockquote>`, never a list/toggle/table/media) — it runs `splitParagraphSegments(text)` and fans the resulting segments out the same way. `splitParagraphSegments` (in `helpers/`) strips surrounding quotes/guillemets, splits on sentence enders `.!?`, falls back to commas/semicolons when there's only one sentence, requires ≥2 segments, and skips (returns `[]`) past `MAX_PARAGRAPH_SEGMENTS` (25) so long prose isn't shredded. v1 is page-level single-paragraph only — a toggle's answer paragraph and multi-block pages are never auto-split. (4) `DeckParser.handleHTML` via `buildPageLinesOverlappingNotes(dom)` for a **page that is many short one-line blocks** (a song, poem, or recited quote where each line sits in its own `<p>`/`<blockquote>`, often wrapped in a `display:contents` div) — `pageBodyLooksLikeLines(dom)` gates it, and `extractTextLines` (in `helpers/`) sibling-walks `.page-body`, unwraps one `display:contents` level, takes each inner `<p>`/`<blockquote>` text, and drops empty verse-break paragraphs. The guard fires only when every non-whitespace `.page-body` block is a `<p>`/`<blockquote>` (no `ul/ol/details/table/h1..h6/img/iframe/figure`), there are ≥2 non-empty lines, none contain `{{c`, and every line is short (≤80 chars) with no mid-line sentence punctuation and no sentence-ending trailing period — so genuine prose paragraphs stay basic while lyric lines fan out. A trailing `!`/`?` or a closing bracket/quote right after sentence punctuation (e.g. `(Kom igjen gutta!)`) is allowed. Order in `handleHTML`: list detection runs first, then single-paragraph detection, then lines detection — single-paragraph must win for a sole `<p>` so its sentence/clause splitting is preserved; lines is the last fallback. The two list paths don't double-process: a page-list note carries `back=''`, so `buildOverlappingClozeNotes` finds no items. Lists under 2 items fall through to the normal cloze path silently. `extractListItems` collects items across **consecutive sibling list blocks of the same region** — real Notion exports emit a numbered list as many single-`<li>` `<ol start="N">` blocks (each wrapped in a `<div style="display:contents">` that `flattenDisplayContentsElements` unwraps before extraction), not one `<ol>` with N `<li>`; it stops merging at the first non-list sibling so two genuinely separate lists divided by a heading or paragraph stay separate. Never branch inside `handleClozeDeletions` for this.
- `helpers/extractHeadingFromSummary.ts` is the small pure helper used by `removeNestedTogglesLegacy` to preserve toggleable-heading content (h1–h6) when stripping a nested `<details>` block from a toggle body. Without it, `<details><summary><h3>Key point</h3></summary>body</details>` collapsed to an empty string and the heading text disappeared from the card back. Only `<h1>`–`<h6>` markup is preserved — plain summary text in a nested toggle is still removed (that's the question, which we don't want duplicated on the back). New-format toggles use a different path (`preserveNestedTogglesBeforeFlattening`) that already preserved nested-toggle summary HTML; the helper is for the legacy export shape.
- `experimental/` is exactly that — gated behind feature flags. Don't fold it into the main path until the flag is removed.
- `Settings/` reads per-user parser overrides from `ParserRulesService`. New options need both a setting and a sensible default.
- **Card-option defaults have one source of truth: `Settings/cardOptionDefaults.ts` (`CARD_OPTION_DEFAULTS`).** `CardOption.LoadDefaultOptions()` returns that map (the Notion/default path), and `controllers/CardOptionsController/supportedOptions.ts` derives each catalog `defaultValue` from `CARD_OPTION_DEFAULTS[key] === 'true'` so the settings UI shows exactly what conversion does. A new boolean option must add its key to the map; `Settings/cardOptionDefaults.test.ts` locks every catalog boolean to the map. The constructor's per-key absent-fallbacks (e.g. `useTags`/`isCloze`/`embedImages` default on when absent) are the raw-upload path and are intentionally distinct from the map — do not unify them. `enable-input` is absent→off (`=== 'true'`): a raw upload with no settings does not turn bold text into type-in fields.
- `getFileContents.ts` is the only conversion-path file in this dir that touches the filesystem at request time. `canary/scheduleParserCanary.ts` reads fixture files at job time — that is intentional and isolated to the canary subdirectory.
- **Cheerio in every markdown path uses fragment mode, not xmlMode.** `embedImagesInHtml` (called from `applyHeuristic`) and `handleNestedBulletPointsInMarkdown` (the default-on nested-bullet path) both walk markdown-it-rendered HTML to rewrite `<img>` srcs. `cheerio.load(html, null, false)` preserves named HTML entities like `&nbsp;` correctly. `xmlMode: true` mangles them — `&nbsp;` round-trips out as the numeric reference `&#xa0;` (and named entities markdown-it leaves intact double-escape to `&amp;nbsp;`), rendering as broken or visible entity text on card backs. The heuristic-path bug shipped on 2026-05-05 and was reverted; the nested-bullet path carried the same xmlMode load until #3234. Load card-back HTML in fragment mode on both paths — never `{ xmlMode: true }`.
- **Notion S3 presigned URL fallback.** Both `embedImagesInCardContent` (toggle/HTML path) and `embedImagesInHtml` (markdown heuristic path) now handle `<img src="https://prod-files-secure.s3…">` URLs that would otherwise be skipped by `isImageFileEmbedable`. When a ZIP entry whose basename matches the URL filename is present, the image is read from the local ZIP entry instead of the expired S3 URL. The helper `exporters/resolveNotionS3ImageFromZip.ts` is the lookup function; the Notion Live API path (via `BlockHandler`) fetches fresh URLs and is unaffected.
- **Host-glued Notion export image recovery.** Some Notion HTML/ZIP exports emit an `<img src>` where a Notion host is concatenated directly onto the relative subfolder path with no separator slash (e.g. `https://app.notion.comObstetrics 1/image 1.png`). That string is not a valid URL — it can neither be parsed with `new URL()` nor fetched (the SSRF guard throws `[observability] invalid URL`), so the image would be silently dropped. `embedLocalOrZipImage` falls back to `exporters/recoverNotionExportImageFromZip.ts`, which strips the bogus Notion host prefix (`app/www`.`notion.com|so|site`) and matches the recovered relative path or basename against a ZIP entry, embedding the local bytes instead of dropping the image. The file is present in the export, so the image survives. Only the host-glued corruption (no leading slash after the host) is recovered; a well-formed `https://app.notion.com/path` is left for the normal remote path.
- **Remote image rehosting.** When an `<img src="http(s)…">` is neither a local/zip path nor resolvable from a ZIP entry (a signed Notion S3 URL whose bytes aren't in the export, or a real external URL the user pasted into Notion), `embedImagesInCardContent` fetches the bytes via `downloadMediaOrSkip` (SSRF-guarded `instrumentedAxios`), bundles them under a deterministic `<sha512>.<ext>` filename, and rewrites the `<img src>` to that local name so the card renders offline and never 403s when the signature lapses. A fetch failure leaves the original URL untouched — never worse than before. This makes the embed phase (`embedImagesInCardContent` → `embedCardImages` → `transformCard` → `processPayload` → `build`/`writeDeckInfo`) async; all are awaited by their callers.
- **Embed-images opt-out.** `CardOption.embedImages` (key `embed-images`, default `true`) gates all three image-embed paths: `applyHeuristic` (markdown heuristic), `embedCardImages` (toggle/HTML), and `buildNoteFromBack` in `handleNestedBulletPointsInMarkdown`. When `false`, the parser skips the embed entirely — image bytes are never copied into media and cards keep their source `<img>` references as-is. Use case: an upload is over the size cap because of images, and the user would rather ship a leaner deck than split the page. The corresponding Notion-Live gate lives on `BlockHandler.embedImage` (see `services/NotionService/FEATURE.md`).
- **Section-scoped tags (upload path only).** Three tag scopes exist: per-card `<del>` (handled in `locateTags`), page-global `.page-body > p > del` (`extractGlobalTags`), and the new section scope. Opt-in via `CardOption.useSectionTags` (key `section-tags`, **default `false`**) and only active in **cherry mode** (the only mode where descendant toggles become distinct cards). When on, `collectSectionTags.collectAndStripSectionMarkers(dom)` runs once at the top of `extractCards`: every `<del>` whose nearest enclosing `ul.toggle` is some toggle T is recorded against T (its tags) and stripped from the DOM (its wrapping `<p>` is removed too when emptied), so the marker never becomes a card or leaks markup. Per card, `ancestorSectionTags(dom, toggle, owners)` unions the markers of every enclosing `ul.toggle` (nested toggles compound) and stores them on `Note.sectionTags`; `locateTags` folds `sectionTags` into `card.tags` (deduped against existing per-card tags) before the unconditional global append. Option **off ⇒ the helper never runs ⇒ byte-identical output**; this is the core safety property. The global selector `.page-body > p > del` does not match a `<del>` inside `details`, so the page-global path is unaffected. The Notion-API path (`BlockHandler`) is out of scope — nested toggles never become separate cards there. Pure tag flow, same `sanitizeTags`, no new sink.
- **`<aside>` tags are stripped before card-boundary parsing and before rendering.** Notion's markdown export wraps callout blocks in `<aside>…</aside>`. `handleNestedBulletPointsInMarkdown` strips those wrapper lines from the raw content before the line loop so they never bleed into card backs. `markdownToHTML` strips them before passing content to `markdown-it` so they never appear as escaped `&lt;aside&gt;` text in rendered card HTML. Only the opening/closing tag lines are removed; the content inside is preserved.
- This module depends on `lib/anki/` for the lower-level Anki primitives (deck filename, cardgen, sanitization). Don't duplicate those helpers here.
- **Font-size override (`font-size` key on `CardOption`).** `withFontSize(style, fontSize)` appends `* { font-size:Xpx }` to the note-type style. The value flows into generated CSS, so it is validated against a strict pattern (1–3 digit number, optional 2-decimal fraction, optional `px`/`em`/`rem`/`%` unit) and bounded to 1–100 before emission; anything else (including a CSS-injection payload) is rejected and emits nothing. Both `'20'` (the web range slider's unitless reset) and `'20px'` (the legacy default) are treated as the no-op default and emit no rule, so existing decks keep the template's own size. The web picker (`web/src/components/FontSizePicker.tsx`) sends a bare integer 10–100.
- **Text-color override (`text-color` key on `CardOption`).** `withTextColor(style, color)` appends `* { color:X }` to the note-type style, mirroring `withFontSize`. The empty value (`''` = "Default") emits no rule at all so text inherits Anki's theme and stays legible in light and night mode — Default must never silently mean black. `textColorSwatches.ts` is the single source of truth: the 8-entry `TEXT_COLOR_SWATCHES` list and `isValidTextColor`. Because a user-supplied colour flows into generated CSS, `withTextColor` only emits a rule when the value matches one of the known swatch hexes — any other value (including a CSS-injection payload) is rejected and emits nothing. The web picker (`web/src/components/TextColorPicker.tsx`) renders from a matching swatch list; the two lists must stay in sync.
- **Text-alignment override (`text-align` key on `CardOption`).** `withTextAlign(style, align)` appends `* { text-align:X }` after `withTextColor`, mirroring the same shape. The empty value (`''` = "Default") emits no rule at all, so every existing deck keeps the template's own alignment — Default is a true no-op and must never change existing output. `textAlignOptions.ts` is the single source of truth: the 4-entry `TEXT_ALIGN_OPTIONS` list (`'' / left / center / right`) and `isValidTextAlign`. The value flows into generated CSS, so `withTextAlign` only emits a rule when the value is `left`, `center`, or `right`; anything else (including a CSS-injection payload) is rejected and emits nothing. The web picker (`web/src/components/TextAlignPicker.tsx`) renders from a matching option list; the two lists must stay in sync.
- **Code theme (`code-theme` key on `CardOption`, default `github`).** `validateCodeTheme` allowlists `github`, `one-dark`, `solarized`, `dracula`; any other value falls back to `github`, so a user-supplied string never builds a filesystem path. `getCodeThemeCss(name)` in `src/templates/helper.ts` reads the matching token-only fragment from `src/templates/hljs-themes/<name>.css` (each carries light token rules plus a `@media (prefers-color-scheme: dark)` block — modern Anki webviews honor it). The structural `pre code.hljs` container rules stay in `notion.css`; per-theme token colors live in the fragments. CSS assembly appends `getCodeThemeCss(codeTheme)` after `NOTION_STYLE` in both `DeckParser.handleHTML` and `BlockHandler.buildDeckStyle`. The API path colors via `highlightCode(code, language)` per block; the upload/HTML-export path has no language (Notion's export drops it), so `applyCodeBlockContainer` walks `pre > code` and runs `highlightCode(text)` with no language — same `<code class="hljs">` wrapper and themed container, but no per-token colors. `<span class="hljs-*">` survives `ApkgPreviewService/sanitize` (span + class are allowlisted).
