# NotionService

Wraps the Notion SDK (`@notionhq/client`) and the OAuth dance. Owns token storage hand-off, top-level-page caching, and block walking for the converter.

## Responsibilities

- **OAuth:** `getNotionAuthorizationLink` and the redirect handler in the controller layer. Tokens land in `INotionRepository`.
- **Top-level pages cache:** `topLevelPagesCache` + `topLevelPagesRefreshGate` — pages are stale after `TOP_LEVEL_PAGES_STALE_AFTER_MS` (5 minutes). The gate prevents thundering-herd refreshes when many tabs hit the picker at once.
- **Block walking:** `BlockHandler/` and `blocks/` recursively fetch a page's blocks and hand them to the parser. Pagination, image URL refresh, and rate-limit handling live here.
- **Per-page settings on recursion:** `BlockHandler` accepts an optional `(settingsRepository, owner)` pair via its constructor (`(exporter, api, settings, settingsRepository?, owner?)`) — it holds the repository interface, not knex. Before walking each page, `findFlashcardsFromPage` resolves that page's own settings row (`loadIfExists(owner, pageId)`) and swaps `this.settings` for the subtree, restoring the parent's in a `finally`. A child page with its own row overrides `deckName` and inherits every other field from the nearest ancestor that has a row; a child with no row inherits the parent's settings entirely. `mergeChildPageSettings` (in `BlockHandler/helpers/`) is the single place that merge lives. The owner is the authenticated job owner, never attacker-controllable, so a job only ever reads that owner's rows. Without the repo/owner injected (e.g. non-job callers) it falls back to the single top-level settings — no per-page resolution. Tag overrides per child are out of scope: tags live in `ParserRules`, not `CardOption`.
- **Positional heading tags (`ParserRules.TAGS === 'heading'`):** when a user enables heading tagging, each card is tagged with the nested ancestor heading chain that precedes it — `H1::H2::H3` (Anki hierarchy). `buildHeadingTagMap` (in `BlockHandler/helpers/`) is the pure source of truth: given the full expanded block list (built upstream in `collectDecksFromPage` from `blocks` after `expandSyncedBlocks`, before the card-only filter — headings are gone by `getFlashcards`) and the same card classifier the cBlocks filter uses, it returns a `Map<blockId, string>` of the heading chain per card block. It tracks a per-level context (H1/H2/H3 only — H4 ignored) as it walks: a heading at level L sets `context[L]` and clears deeper levels; a card snapshots `[H1,H2,H3].filter(Boolean).join('::')`. A toggle-heading that is itself a card is tagged by its ancestors, not its own level (chain is snapshotted before the heading's own level is applied). Each heading title becomes one safe segment via `sanitizeTags` (spaces→hyphens), joined with `::`; the joined string is never re-tokenized afterwards, so a title literally containing `::` stays intact. The map is threaded into `getFlashcards` and applied per card by block id, added to existing tags (deduped via `Set`, never clobbering). This **supersedes** the old page-global behavior: `getTopLevelTags` now returns `[]` for the heading case (it used to collect every heading on the page and apply the whole list to every card). The strikethrough/global-paragraph tag path in `getTopLevelTags` is untouched.
- **Heading fields (`CardOption.template === 'hierarchy'`):** when the user picks the Hierarchy card style, each basic card carries its ancestor headings as note fields, not just tags. `buildHeadingContextMap` (same file as `buildHeadingTagMap`, which is now a thin derivation over it — one traversal implementation, two output shapes) returns `Map<blockId, { h1?, h2?, h3? }>` with **raw trimmed heading text** (spaces preserved — fields are display text, only tags hyphenate). `collectDecksFromPage` builds it from the same expanded block list and card classifier as the tag map and threads it into `getFlashcards` as a separate parameter; `applyHierarchyFields` sets `note.hierarchy = true` plus `h1/h2/h3` (missing levels → `''`) on basic-type notes and table-row notes only — cloze, MCQ, and input notes keep their own models. On the Python side `create_deck.py` routes `card.hierarchy` to the `n2a-hierarchy` model (`src/templates/n2a-hierarchy.json`: H1, H2, H3, Question, Answer, MyMedia; breadcrumb front via `{{#H1}}…{{/H1}}` conditionals). Heading tags and heading fields are independent — `TAGS=heading` and `template=hierarchy` can be on together or separately. Non-Notion uploads with `template=hierarchy` fall back to plain basic cards (no heading source).
- **Database preview (read-only):** `NotionAPIWrapper.queryDatabasePreview(id, pageSize)` returns the first `pageSize` rows from the first data source only and a `hasMore` boolean — no aggregation across data sources, no pagination, no total-row count (Notion's `dataSources.query` doesn't return one). Used by `GetDatabasePreviewUseCase` (in `src/usecases/notion/`) to power the `/preview/database/:id` page and the `GET /api/notion/database/:id/preview` endpoint. The endpoint runs `inferColumnMapping` server-side and ships a typed `DatabasePreview` payload (`title`, `url`, `columns`, `mapping`, `samples` with `{ id, values }` shape, `rowCount`, `hasMore`) — no raw Notion property objects reach the client. The use case extracts cell text via a dispatch table covering `title`, `rich_text`, `select`, `multi_select`, `status`, `number`, `checkbox`, `url`, `email`, `phone_number`, `date`, `created_time`, `last_edited_time`, `people`, `files`, and `formula` properties. The picker for ambiguous mappings stays on `/downloads` — preview is read-only.
- **Database as flashcards (row-as-card):** `BlockHandler.findFlashcards` routes `parentType === 'notion-database' | 'database' | 'data_source'` to `findFlashcardsFromDatabaseRows`, which queries the database, auto-infers the front/back column mapping via `lib/notionDatabase/inferColumnMapping` (canonical names: Term/Word/Front/Question/Vocabulary on the front; Definition/Meaning/Back/Answer/Translation on the back), and emits one Anki note per row. Callers may pass explicit `frontField` / `backField` on the `Finder` to bypass auto-infer (used by the picker UX). When the mapping is ambiguous and no explicit fields are provided, the conversion path no longer hard-fails: the method degrades to the first column as front and the second as back, records `guessedColumnMapping: { frontField, backField }` on the handler, and continues — a usable deck instead of a `NOTION_DATABASE_COLUMNS_AMBIGUOUS` job failure. `guessedColumnMapping` rides through `CompleteJobUseCase` into the `notion_columns_guessed` conversion signal so the user is told which columns were guessed and can convert again to pick. The `NOTION_DATABASE_COLUMNS_AMBIGUOUS` error code and its `jobFailureReason` handling stay for the preview/picker path (`inferColumnMapping` still reports `ambiguous`); only the row-as-card conversion stopped treating it as fatal. Only `title` and `rich_text` property types are read in v1. Rows with empty front or back text are skipped. The legacy "recurse into each row as a sub-page looking for toggles" branch is gone — it only ever processed the first row anyway, and the row-as-card path is the supported model now. **Page-link self-heal:** when `parentType === 'page'` but the link is actually a database (the one-shot converter defaults `parentType` to `page` for every Notion link, so a pasted database URL lands here), `findFlashcardsFromPage` throws Notion's `validation_error: … is a database, not a page`; `findFlashcards` catches that via `helpers/isNotionDatabaseNotPageError` and falls through to `findFlashcardsFromDatabaseRows` so the deck still builds and takes the database's title. The same helper backs the Ankify sync fallback (`SyncNotionPageToRacUseCase.walkSource`), the favorites enrichment self-heal (`GetEnrichedFavoritesByOwnerUseCase`), and the `notion_database_not_page` job-failure reason — one definition, four call sites.
- **Free-plan truncation accounting (visibility only — no limit change):** free conversions fetch one Notion page of blocks (`getBlocks` requests `DEFAULT_PAGE_SIZE_LIMIT = 200`, but Notion clamps `page_size` to 100) and only paginate when `all === true` (`bl.useAll = isPaying` in `CreateJobWorkSpaceUseCase`). `helpers/conversionTruncation.ts` is the single source of truth for detecting that: `isTruncatedBlockFetch(all, response)` is `!all && response.has_more` — never read `response.has_more` alone, because the paid pagination loop exhausts the cursor without resetting `has_more`. `BlockHandler.recordTruncation` accumulates `truncation: { blocksConverted, subDeckRulesSkipped }` across page-level fetches only (back-side/table/sub-block fetches are excluded so the count means "blocks of the page"); `subDeckRulesSkipped` flags `hasRuleBasedSubDecks` (any `SUB_DECKS` entry besides `child_page`, which traverses on every plan). `performConversion` hands `bl.truncation` and `bl.droppedAssetCount` to `CompleteJobUseCase`, which serializes `buildNotionConversionSignalPayload` into `jobs.job_reason_failure` on done — one JSON object carrying the non-fatal signals (`{ code, blocks_converted?, sub_deck_rules_skipped?, dropped_assets?, front_field?, back_field? }`; `code` is `notion_truncated` when the page truncated, else `notion_assets_dropped` when assets dropped, else `notion_columns_guessed` when a Notion database's columns were auto-guessed), namespaced by `code` so it can't be confused with the `apkg_import` done payload (client parsing for that is gated on job type). Truncation and dropped-assets co-occur and share the single column, so the builder merges them rather than letting two codes clobber each other; the guessed-columns signal comes from the database row-as-card path, which never truncates or drops assets, so it takes the column on its own. `BlockHandler.droppedAssetCount` increments at each `embedImage`/`embedAudioFile`/`embedFile` arm where the download returned null (attempted-and-failed only — an `external` link never reaches that arm). The Downloads page parses the column three ways: `parseTruncationPayload` (gated on `page`/`database` types) renders the expandable "Partial" note with the upgrade link, `parseDroppedAssetsPayload` reads `dropped_assets` regardless of `code` and renders the "Some images skipped" warning chip + `ImageDropNotice`, and `parseColumnsGuessedPayload` reads `front_field`/`back_field` and renders `ColumnsGuessedNotice` telling the user which columns were guessed. The upload/worker conversion path and the Ankify poll-sync path do not yet carry this count — Notion one-shot conversion only (#3457 part 1).
- **Toggle-less fallback (heading+content / Q:A / term::definition):** when a page's own card walk produces zero cards (`getFlashcards` returns `[]` in `collectDecksFromPage`), `guessTogglelessCards` runs `helpers/guessCardsFromBlocks` before the deck is built. It flattens the already-walked `blocks` into a plain-text markdown document (`heading_1/2/3` → `## text`, other rich-text-bearing blocks → their plain text; blocks with no `rich_text` such as dividers and child pages are skipped) and reuses the upload path's `lib/parser/guessMarkdownCards` heuristic so the live-Notion and upload paths make the same cards for equivalent content (heading paired with following content, `Q:`/`A:` paragraphs, `term::definition`). Only heading+content and Q:A/inline shapes are covered — no nested structure, tables without headers, or rich formatting on the back (plain text). When the heuristic also finds nothing (prose with no card shape) the deck stays empty and the existing `EmptyDeckError` "wrap your key terms in toggles" message still fires — the fallback is strictly additive. Guessed cards feed `cardCount`/`emptyBackCount` like normal cards. This closed the largest live-Notion failure category (toggle-less pages hard-failing) vs the upload path, which already had `guessMarkdownCards`.
- **Unknown parentType fallback:** any unrecognised `parentType` value logs `[notion] unsupported parentType: <type>` via `console.error` and returns an empty `Deck[]` rather than throwing. The caller already handles empty decks via `EmptyDeckError`; this prevents a single unknown parent type from crashing the whole conversion job.
- **MCQ classification:** when `settings.mcqEnabled` is true, `BlockHandler.findFlashcards` runs `detectNotionApiMCQ` (from `lib/parser/findNotionToggleLists`) against the rendered back HTML of each toggle. If the children render as `<ul class="to-do-list">` with exactly one `.checkbox-on`, the note becomes an MCQ note (mcq=true, options + correctIndices populated, back cleared). When `mcqEnabled` is off the path is a no-op. MCQ takes precedence over cloze and input detection on the same block.
- **Table blocks:** when `table` is in `flashcard_is`, `BlockHandler.getFlashcards` handles each `table` block with an early branch: it fetches `table_row` children once via `api.getBlocks`, then calls `tableRowsToCards` (from `blocks/lists/BlockTable.tsx`) to produce `{front, back, isCloze}` triples — one per data row. Column roles default to col1 = front, col2 = back; when `has_column_header` is true, `resolveColumnRoles` reads the header row's cell text and runs it through `inferColumnMapping` (the same name heuristic the database picker uses — `term`/`word`/`front`/`question`/`vocabulary` vs `definition`/`meaning`/`back`/`answer`/`translation`, case-insensitive) so a header-labelled but column-reversed table (e.g. `Definition | Term`, `Back | Front`) maps front/back to the right columns. The fallback is byte-identical positional behavior: no header row, or a header whose names don't map cleanly to a distinct front and back, keeps col1 = front, col2 = back. Rows where the resolved front or back column is empty are skipped. When `settings.isCloze` is on (the default) and a rendered cell contains inline `<code>` outside `<pre>`, the row becomes a cloze note via the same `hasInlineClozeCode` + `handleClozeDeletions` pair the toggle-content path uses: code in col1 → col1 becomes the cloze Text and col2 stays the back; code only in col2 → col2 becomes the cloze Text and col1 becomes the back. `BlockHandler.getFlashcards` copies `isCloze` onto `Note.cloze`. 1-column tables produce no cards. Every column other than the resolved front and back appears, in original order, as an inline `<table>` appended to the back (for a positional 2-col mapping that is cols 3..N). When `has_column_header` is true the first row is skipped from card seeding. `BlockTable` (the back-body renderer) also handles the case where a `table` block appears inside an existing card's back-body — it emits a full `<table>` with optional `<thead>` when the column header flag is set. Standalone `table_row` blocks render to an empty string in `blockToStaticMarkup` — their cells are owned by the parent `table`, so if a walker ever hands one through directly (e.g. children iteration after the table has already rendered) we drop it instead of leaking `unsupported: table_row` + JSON to the user.
- **File uploads:** `NotionAPIWrapper.uploadFile` uses the Notion file uploads API (`fileUploads.create` + `fileUploads.send`) to upload media (images) to Notion's storage. Returns a `file_upload` ID that can be referenced in image blocks. Used by the APKG import flow.
- **Search normalisation:** `NotionAPIWrapper.search(query)` paginates Notion's `/v1/search` (following `next_cursor` until `has_more` is false, capped at `SEARCH_MAX_PAGES = 20`) and aggregates every page's results before running them through `helpers/collapseDataSourcesToDatabases`. Notion returns at most 100 results per page; without pagination a workspace with many title matches drops every match past page 1 from the `/notion?q=` picker (a sibling page sorted after the first 100 by `last_edited_time` would silently disappear). The method takes no `all` flag — pagination is unconditional for the single search surface that calls it (`NotionController.search` → `Backend.search`). After aggregation: Notion's 2025-09-03 API splits each database into a wrapper `database` plus one or more `data_source` children, and unfiltered search returns the `data_source` (not the wrapper). The helper rewrites `object: 'data_source'` to `object: 'database'` and substitutes the parent `database_id` as the result `id`, so downstream code (`Backend.search` in the web app, `getDatabase` on click, `databases.retrieve` in the conversion job) keeps working against the database wrapper ID. Sibling data_sources of the same database are deduped. Data_sources with no `parent.database_id` are dropped (they can't be opened as a database). Notion's 2026-04-22 change made search cursors session-scoped and opaque, so a `next_cursor` handed out early can be rejected with a `validation_error` referencing `start_cursor` before the 20-page loop finishes. `helpers/tolerateCursorInvalidation.ts` (`fetchPageOrStopOnCursorInvalidation`) wraps every cursor-forwarding pagination loop in the file — `search`, `queryDatabase` (`dataSources.query`), `searchTopLevelPages`, and the `getBlocks` deep-fetch — so a mid-loop cursor invalidation (a cursor is already set, i.e. not the first page) breaks the loop and returns the results collected so far instead of discarding them and surfacing Notion's raw error. Only that precise error is swallowed (`APIResponseError`, code `validation_error`, message contains `start_cursor`); a first-page invalidation and any other validation error still propagate. A truncated `getBlocks` fetch is never written to the block cache, so the incompleteness is transient rather than sticky.
- **Top-level picker surfaces databases too:** `NotionAPIWrapper.searchTopLevelPages(query, opts)` powers the Ankify "Search your Notion pages" and converter "Where should we put it?" pickers. It runs two Notion searches: the page pass (`filter.value = 'page'`, dropping database rows and untitled pages) and a second `searchTopLevelDatabases` pass (`filter.value = 'data_source'`) whose results go through `collapseDataSourcesToDatabases` and are appended as entries with `object: 'database'`, keyed on the parent `database_id`. Without the database pass a shared database never appeared in the picker, so a user studying from a database (pages-with-toggles or row-as-card) couldn't subscribe it to Ankify and had to paste the link. The result `object` widened from `'page'` to `'page' | 'database'`; the front-end `NotionObject.object` is already `string`, so the picker labels and icons databases with no UI change. Databases are appended after pages and are not persisted to the tier-2 `top_level_pages` cache (page-only) — they come fresh each search.
- **Identity helpers:** `getNotionId`, `isValidNotionId`, `NotionColors`.
- **Notion highlight color mapping (`NotionColors.ts`).** `NOTION_COLORS` is the single source of truth: 10 text colors (`default`, `gray`, `brown`, `orange`, `yellow`, `green`, `blue`, `purple`, `pink`, `red`) plus 9 background variants (every text color except `default` has a `<name>_background` entry). `notionColorToHex(name)` resolves the token to the hex value; unknown tokens fall back to `default` (`#37352F`). `isNotionColorBackground(name)` is the `endsWith('_background')` check used by `HandleBlockAnnotations.tsx` to decide between inline-style background and CSS class. `styleWithColors(color)` returns `' n2a-highlight-<token>'` — preserved verbatim, never lowercased — so the corresponding `.n2a-highlight-<token>` rule in `src/templates/notion.css` (and the generator in `scripts/generate-notion-color-css.js`) wins. `default` is intentionally inert so plain text doesn't get a redundant span. **When adding a new color or renaming a token, update three places in one commit:** `NotionColors.ts` (the lookup), `src/templates/notion.css` (the rendered swatches Anki ships), and `scripts/generate-notion-color-css.js` (the regenerator). Mismatch shows up as a card with the wrong color in Anki but the right hex in the converter logs.
- **Linked toggle (synced_block) expansion:** `helpers/expandSyncedBlocks.ts` flattens every `synced_block` in a block array into its resolved children — for an original (`synced_from = null`), the children come from `getBlocks(<synced_block.id>)`; for a reference, from `getBlocks(<synced_from.block_id>)`. `BlockHandler.findFlashcardsFromPage` runs this on the page-level fetch (and the `useAll` subDeck branch); `getBackSide` runs it on every fetched children array so synced blocks nested under a toggle render their source content inline. Without expansion, `classifyBlock` returns `'skip'` for `synced_block` and a toggle wrapped in a linked block silently disappears from the deck. Cycle detection via a `Set<string>` of resolved source IDs; `MAX_DEPTH = 8` caps recursion on deeply nested synced trees. `blockToStaticMarkup`'s `case 'synced_block'` is the defense-in-depth — for any synced block that slips past expansion it renders the resolved children inline (via `getBackSide` when `has_children`), falling back to an `<aside class="synced-block-empty">Synced block</aside>` placeholder, rather than the JSON `unsupported` dump.
- **PDF and link-preview blocks in `blockToStaticMarkup`:** `case 'pdf'` renders an `<embed type="application/pdf">` pointing at `pdf.file.url` (Notion-hosted) or `pdf.external.url`, followed by a fallback `<a>` to the same URL. `case 'link_preview'` renders a single `<a href rel="noopener noreferrer">` showing `link_preview.url` as the link text. Both mirror the sibling `renderBlockPreview.ts` cases and rely on React's `renderToStaticMarkup` for attribute/URL escaping (the file is `.tsx`).
- **heading_4 and the unknown-block fallback in `blockToStaticMarkup`:** `case 'heading_4'` renders through `BlockHeading('heading_4', …)`, which folds to `<h3>` markup — the same h4→h3 fold `renderBlocks.ts` uses; do not invent an h4 visual level. `getHeading`, `getHeadingColor`, `isToggleHeading`, and `getToggleSummaryRichText` all carry the `heading_4` arm, so toggleable heading_4 behaves exactly like heading_3. The `default` arm renders **nothing**, calls `handler.recordUnsupportedBlockType(c.type)` (tallies an occurrence per type in `BlockHandler.unsupportedBlockTypeCounts`, a `Map<string, number>`, mirroring `droppedAssetCount`; the `unsupportedBlockTypes` getter expands the counts back to a flat array so a pathological doc never holds a large duplicate-filled array during the walk), and logs `console.debug('unsupported <type>')` — never reintroduce the old `unsupported: <type>` placeholder or the pretty-printed block JSON, which leaked block IDs and created_by user IDs onto study cards. After a successful conversion, `performConversion` fires `bl.unsupportedBlockTypes` at `UnsupportedNotionBlockRepository.record` (fire-and-forget, never blocks or fails the job) so the `/ops` dashboard can surface which new Notion block types are showing up in real conversions. Genuinely contentless types (`table_of_contents`, `breadcrumb`, `template`) are intentionally invisible on cards. `BlockHandler` also accumulates `cardCount` and `emptyBackCount` (empty-back cards produced before `Deck.CleanCards` drops them, via `countEmptyBacks`) at the single `getFlashcards` return; `performConversion` records `{ decks, cards, emptyBack }` to `ConversionOutputStatsRepository` under source `convert` so silent empty-back loss becomes an ops signal.

## Calling pattern

Every outbound call must go through `instrumentedAxios` (from `services/observability/`) or the wrapped `NotionAPIWrapper.ts`. Do not import `axios` or call `notion.request` directly from this dir.

`NotionAPIWrapper`'s constructor takes `(key, owner, blocksCache?)`. The optional `blocksCache: IBlocksCacheRepository` (from `data_layer/BlocksCacheRepository.ts`) backs `getBlocks` — when present, `all`-mode reads check the cache before hitting Notion, and writes persist the response. Pass it in production via `NotionService`'s constructor; omit it in tests for a no-op (no caching, no DB access). Never import `getDatabase` here.

## Things to know before editing

- **Authorize URL needs `redirect_uri`:** `getNotionAuthorizationLink` must include the `redirect_uri` query param (sourced from `NOTION_REDIRECT_URI`). Notion rejects the authorize request with "Missing or invalid redirect_uri" if it's absent. The companion `redirect_uri` on the token exchange (`oauth/token`) is separate but must match.
- **Native return state carries the app owner, HMAC-signed:** `GET /api/notion/get-notion-link?client=native` threads a signed `state` into the authorize URL — `getNotionLinkInfo(owner, { client: 'native' })` calls `buildNativeOAuthState(owner, process.env.SECRET)` (in `nativeOAuthState.ts`), producing `native:<owner>:<ts>:<hmac>`. The owner is the **app's** authenticated user at link-mint time, not the browser's. At the callback, `NotionController.connect` routes any state matching `isNativeOAuthState` to `connectNative`, which calls `verifyNativeOAuthState(state, process.env.SECRET)` to verify the HMAC (timing-safe) and the 300s freshness window, then ties the token to the **decoded owner** — never `res.locals.owner` (the browser session). On verification failure it redirects to `twoanki://x-callback-url/notion-connected?error=invalid_state`; on success to the bare deep link `twoanki://x-callback-url/notion-connected`. **Why signed:** state round-trips through the browser and Notion and is fully user-controllable; a raw owner id would let anyone attach a token to any account. The deep link stays the single server-held constant — the client cannot pass a redirect target. A legacy bare `state=native` (old app build, no owner) verifies to `null` and is rejected with `?error=invalid_state` — attributing it to the browser session is exactly the bug we fixed (#2999). No `state` → identical web URL and `/notion` redirect as before. Keep the native check after the `login:` branch so login flow ordering is undisturbed.
- **Cache invalidation:** writes (e.g. user disconnects, token refresh) call `invalidateTopLevelPagesForOwner`. Forgetting this leaves stale pages visible to other tabs for up to 5 minutes.
- **Refresh gate:** `tryClaimRefresh` returns false for the second concurrent caller — that caller should serve stale data, not block. Don't change this to a mutex without thinking about the perf tail.
- **Webhooks:** the receiver in `routes/AnkifyWebhookRouter.ts` is intentionally inactive; see `Documentation/ankify/notion-webhooks-deferred.md`. Polling at 5 min carries the near-realtime story today.
- **Token hashing:** never log raw tokens. Use `hashToken` from `lib/misc/hashToken` for any audit trail.
- **Transient-error retry:** every Notion SDK call in `NotionAPIWrapper` is wrapped in `helpers/withRetry`, which retries transient failures with bounded exponential backoff + jitter (3 attempts default). Retryable: `APIErrorCode.RateLimited` (honoring the `Retry-After` header, capped at 30s), `InternalServerError`/`ServiceUnavailable`/`GatewayTimeout`, the client-side `RequestTimeoutError`, an `UnknownHTTPResponseError` whose `status` is a transient gateway code (429/500/502/503/504), and the network codes (`ECONNRESET`, `ETIMEDOUT`, …). Non-transient codes (`Unauthorized`, `ObjectNotFound`, `ValidationError`, 4xx) throw immediately. Only after retries are exhausted does the error surface — at which point a user-facing message ("Notion is busy, retry in a moment") is appropriate; do not add a second retry layer on top.
- **Dead-token flag:** `notion_tokens.invalidated_at` (nullable timestamptz) is set on the first `Unauthorized` response from Notion. `getNotionLinkInfo` treats a flagged row as `isConnected: false` so the existing reconnect affordance reappears. Call `INotionRepository.markTokenInvalid(owner)` at every 401 call site (controller search, poll worker), and `clearTokenInvalid(owner)` on reconnect. Never hard-delete the row — the soft flag is the safety margin for a mistakenly-flagged live token.
- **Embed-images opt-out.** `BlockHandler.embedImage` returns `''` (no media added) when `settings.embedImages` is false. This is the Notion-Live counterpart to the file-upload path's `CardOption.embedImages` gate (see `src/lib/parser/FEATURE.md`). Use case: user is hitting the upload size cap because of images and wants a leaner deck. Default stays `true` — existing conversions are unchanged. `embedAudioFile` and `embedFile` are intentionally NOT gated; the opt-out is image-only because images are the typical size cap driver.
- **Expired-signed-URL recovery.** Notion serves `type === 'file'` images, audio, and files via a signed S3 `file.url` that expires roughly an hour after generation; on a large page or a long job the signature can lapse before the download. `embedImage` / `embedAudioFile` / `embedFile` route their download through `downloadWithFreshUrlRetry`, which retries exactly once for `file`-type assets: on a `downloadMediaOrSkip` miss it re-fetches the block via `api.getBlock(blockId)` (Notion mints a fresh signed URL on every retrieve), re-runs the same `getImageUrl` / `getAudioUrl` / `getFileUrl` extractor, and retries the download. `external` assets are never re-fetched — their URLs do not expire. The retry is bounded to one and degrades gracefully: a second miss, a null fresh URL, or a failed re-fetch returns null and the existing skip-not-abort behavior stands. `getImageSourceType` / `getAudioSourceType` / `getFileSourceType` (colocated with each URL extractor) supply the `file`-vs-`external` discriminant. No new HTTP client — the second download reuses `downloadMediaOrSkip` (`instrumentedAxios`), so the SSRF guard still covers it.
- **Page-title fallback is the stable string `Untitled`.** When `getNotionObjectTitle` cannot extract a title (untitled page, database row whose title property is empty, e.g. Snipd-imported pages where the user hasn't filled the title), `getPageTitle` returns `'Untitled'` — never a stamped string. The previous behaviour stamped `Untitled: <Date>` into the deck name, which broke Anki's name-based dedupe (every retry produced a new deck) and surfaced raw `new Date()` output in the user's Anki sidebar. Keep the fallback short and deterministic; if you need to disambiguate multiple untitled pages, do it at the deck-naming layer (`getDeckName(parent, …)` already prefixes the parent path).
