# Spec: automatic topic-based tagging on generated cards

## Problem

External review (May 2026) flagged this directly:

> Consider automatic tagging based on topic, where feasible.

Today, cards generated by the Claude pipeline (HTML/markdown → JSON) and by the Photo-to-Deck pipeline (vision → JSON) ship to Anki without tags. Anki's review and filtering UX leans heavily on tags — a user studying biochemistry expects to filter their deck by `enzymes`, `kinetics`, `regulation`. Right now they get one undifferentiated pile.

The plumbing is already half-done. `src/lib/claude/ClaudeService.ts:15` declares `"tags": string[] — topic tags` in the system prompt's JSON schema. `CompactCard.tags?: string[]` (line 94) is typed. `buildDeckInfo` already forwards `tags: c.tags ?? []` to the Python deck builder (line 216). No spec or PR has ever committed to topic-tag behavior end-to-end, so the model is free to omit tags and silently does.

Adjacent issues (#261 tag creation, #437 local tags, #614 group-heading tags, #2452 per-column Notion → tag rules) cover manual or rule-based tagging. None cover AI-generated topic tags.

## Goal

Cards arrive in Anki pre-tagged with 1–3 topic tags so a learner can filter their deck the moment it opens. This traces directly to "beautiful Anki flashcards out of the box" — organized at delivery, not on the user's time.

Success: a converted deck from a multi-topic Notion page or a multi-subject photo lands in Anki with sensible per-card tags, with no UI for the user to learn.

## Non-goals

- Not building a user-facing tag editor in 2anki's UI (issue #261 already tracks that — different surface).
- Not building per-column Notion → tag mapping (issue #2452 is the right home).
- Not exposing a settings toggle. Default-on, no toggle — adding a knob trades clarity for noise that 0.1% of users will discover.
- Not introducing a tag taxonomy or controlled vocabulary. The model picks short, lowercase, snake_case topic tags from the source text. Anki's own tag tree handles the rest.
- Not tagging by source ("from_notion", "from_photo"). Those are metadata, not topics.
- Not modifying the Ankify (Notion → Anki sync) path in this spec. That product has its own loop and gating (`src/lib/ankify/access.ts`); revisit after this lands.

## Proposed shape

**One change to the Photo-to-Deck prompt.** Update `VISION_PROMPT` in `src/usecases/imageOcclusion/PhotoToFlashcardsUseCase.ts:42–51` so the model is asked to emit 1–3 topic tags per card in the same `tags: string[]` field the schema already supports. The JSON output format gains `"tags": string[]` as an optional field, mirroring the Claude-text prompt at `ClaudeService.ts:15`. Tags are short, lowercase, snake_case, drawn from the actual content (e.g. `enzymes`, `michaelis_menten`, `regulation` — not `biology` or `chapter_4`).

**No change to the Claude-text prompt itself** — it already declares the schema slot. What's missing is the *commitment* to the behavior: the implementer should add a test (`ClaudeService.test.ts` or a sibling) that runs a small fixture through the parser and asserts that the parsed cards carry a non-empty `tags` array. Today nothing in the test suite would notice if Claude stopped emitting tags. That fragility is what's letting the field silently stay empty in prod.

**Tag rendering in the `.apkg`.** No change needed. The Python builder consumes `tags: string[]` and writes Anki's native tag field. Anki renders tags in its review screen and tag browser without further work on our side.

**Default-on, no toggle.** Tags add ~10–30 output tokens per card, well under the model's per-card cost. If a user dislikes tags, Anki's bulk-edit lets them strip the field in one click. The cost of asking before adding (a settings page, a toggle, copy, docs) is higher than the cost of removing.

**Edge cases the implementer must handle:**
- Empty / malformed `tags` from the model → coerce to `[]`. `buildDeckInfo` already does this defensively.
- Tags with whitespace, capitalization, or weird punctuation → normalize at the parser boundary (lowercase, replace whitespace with `_`, strip anything outside `[a-z0-9_]`). One small helper next to `CompactCard`.
- Photo input that is genuinely single-topic (a single textbook page) → 1 tag is fine. The prompt asks for 1–3, not exactly 3.
- Pricing/quotas → unchanged. Tags are model output, not a new feature surface.

## Open questions

1. Should normalized tags be capped at a max length (e.g. 32 chars)? Anki's own limit is generous; recommendation is yes, 32 chars, to keep the tag browser readable.
2. Should the Claude-text path also get the test-and-commit treatment in the same PR, or split into a follow-up? Recommendation: same PR — adding the test costs minutes and the schema already promises the behavior.
3. The Ankify (Notion → Anki sync) path goes through the same Claude prompt today. The default-on behavior will affect Ankify subscribers automatically. Do we want to flag this in the next Ankify changelog? Recommendation: yes, one-line changelog entry.
