# Spec: AnKing-style Image Occlusion Enhanced compatibility

### Trio synthesis
- **PM**: Medical students live inside AnKing; if 2anki-emitted IO cards don't share AnKing's note-type shape they can't merge into the user's main study deck, so they sit in a parallel deck and the user goes back to manual authoring. Move IO-deck repeat-usage rate (same user generates ≥2 IO decks in 30 days) up from baseline.
- **Designer**: No new screens. One destination toggle on the Image Occlusion screen — "Note type: 2anki classic / AnKing-compatible" — defaulting to AnKing-compatible for new sessions once the field shape is verified. Existing in-flight sessions keep classic.
- **Engineer**: Add a second note-type template (`n2a-io-anking.json`) and a parallel build path in `create_io_deck.py` that emits `Question Mask` / `Answer Mask` SVG instead of `{{cloze:Occlusion}}`. The current path stays untouched. Effort **M** — the shapes-to-SVG converter is the meaningful new code; everything else is wiring.
- **Agreement**: Ship a second, AnKing-shaped note type alongside the existing one rather than mutating in place. Field names and order must match the AnKing IOE schema exactly (11 fields). Out of scope: auto-detecting occlusion regions (covered by #2614).
- **Conflict**: PM wanted AnKing-compatible to become the default immediately; engineer flagged that the canvas-based renderer in the current `n2a-io.json` is what `ApkgPreviewService` already understands — switching the default risks breaking the in-app deck preview until preview gains a fallback for SVG-overlay templates. Resolved: ship behind a toggle defaulting to classic; flip the default in a follow-up after preview-side work lands.
- **Resulting plan**: Add an `AnKing-compatible` option on the IO conversion screen that emits a deck using AnKing IOE's 11-field note type (`ID`, `Header`, `Image`, `Footer`, `Remarks`, `Sources`, `Extra 1`, `Extra 2`, `Question Mask`, `Answer Mask`, `Original Mask`) and AnKing's SVG-overlay card template. Convert each `OcclusionRect` to an SVG mask; persist per-image group IDs so siblings merge cleanly in the user's collection.

## Spec

**Outcome**: A medical student running AnKing can merge a deck generated by 2anki into their existing collection in one drag — no field-mismatch warnings in Anki's import dialog. Target: lift IO-deck repeat-usage rate (same user generates ≥2 IO decks within 30 days) by 20% within six weeks of launch, measured against the pre-launch month.

**Goal alignment**: Medical students are a core 300K-user lane. AnKing is the de facto curated deck in that lane; cards that don't slot into AnKing don't get used. This makes 2anki's IO output usable, not parallel.

**Problem**: A recent professional Anki/Quizlet creator's usability report called out AnKing compatibility twice: their IO cards must be familiar to medical students and advanced Anki users; they specifically named AnKing-style workflows around Image Occlusion Enhanced as an integration target. 2anki currently emits a 5-field, cloze-based IO note type (`Occlusion`, `Image`, `Header`, `Back Extra`, `Comments`) with a custom canvas script. AnKing IOE uses 11 fields and SVG masks. When a student imports a 2anki IO deck into their AnKing collection, Anki sees a different note type GUID, the field map fails, and the cards land in a sibling deck the student does not study from.

**Riskiest assumption**: That a faithful field/template clone is enough for AnKing users to merge our output into their collection — that we don't also need to match a specific add-on version's model GUID or its CSS variable names. Smallest test: build the spec's note type, generate a deck with 5 occlusion images, import into a vanilla Anki profile with AnKing IOE already installed, confirm cards appear under the existing IOE note type without warnings.

**Scope**:

In:
- New official template `n2a-io-anking.json` with the AnKing IOE 11-field shape and AnKing's SVG-overlay card template (front shows Question Mask, back shows Answer Mask).
- A shapes-to-SVG converter that turns each `OcclusionRect` (including `ellipse` and `polygon`) into a labelled `<rect>` / `<ellipse>` / `<polygon>` inside a single SVG per mask field.
- A `noteType: 'classic' | 'anking'` flag on the IO request, plumbed from the IO upload screen through `CreateImageOcclusionDeckUseCase` and into `create_io_deck.py`.
- UI toggle on the Image Occlusion screen ("Note type"), default value `anking` for new sessions, persisted in user settings the way other IO-screen preferences are persisted today (no new migration if a generic per-user settings store exists; otherwise lift to a new column — see open questions).
- Stable group/note GUID strategy so sibling occlusions on the same image are recognised by AnKing's siblings logic.

Out:
- Auto-detection of occlusion regions from highlighted/bold/underlined terms (PR #2614, `feat/spec-auto-image-occlusion`). This spec is shape-only; #2614 is detection-only.
- AnKing Step Decks compatibility (separate; not the IOE note type).
- A migration path that retroactively rewrites already-generated 2anki classic IO decks into AnKing shape. The user can regenerate if they want the new shape.
- Replacing or removing the existing `n2a-io.json` template. It keeps shipping; the toggle picks between them.

**User story**: As a medical student studying with AnKing, I want 2anki to give me an `.apkg` that merges directly into my AnKing deck, so the cards I just generated show up next to the rest of my AnKing image-occlusion cards instead of in a parallel sibling deck I never open.

**Acceptance criteria**:
- [ ] A new template `src/templates/n2a-io-anking.json` exists with fields in this exact order: `ID`, `Header`, `Image`, `Footer`, `Remarks`, `Sources`, `Extra 1`, `Extra 2`, `Question Mask`, `Answer Mask`, `Original Mask`.
- [ ] Card templates render the image with `{{Question Mask}}` overlaid on the front and `{{Answer Mask}}` overlaid on the back, matching AnKing IOE's wrapper structure (`<div>` wrapper with overlay + image divs).
- [ ] `CreateImageOcclusionDeckUseCase` accepts `noteType: 'classic' | 'anking'` and routes to the appropriate Python builder path; the classic path stays bit-for-bit identical to today.
- [ ] `create_io_deck.py` emits one note per occlusion rect when `noteType == 'anking'` (AnKing's model: one card per mask, siblings grouped by image) rather than one note per image with cloze indices.
- [ ] Each generated note's `Question Mask` and `Answer Mask` fields contain valid SVG with the rect / ellipse / polygon at the original coordinates, scaled to the image's natural dimensions.
- [ ] A generated AnKing-mode deck imports into a vanilla Anki profile with the AnKing IOE add-on already installed and produces no "note type changed" or field-mismatch warning in the import dialog.
- [ ] The IO conversion screen exposes a "Note type" segmented control with values `2anki classic` and `AnKing-compatible`, defaulting to `AnKing-compatible` for new sessions; selection persists per user.
- [ ] Free-tier 3-image limit still applies in AnKing mode.
- [ ] `pnpm test src/usecases/imageOcclusion` plus a new colocated test for the shapes-to-SVG converter all pass; coverage for the new converter ≥ 90%.

**Open questions**:
1. Does the project already have a generic per-user IO settings store, or does AnKing-mode preference need a new column on `users` (or a new `user_preferences` table)? If a new column, decide name and default (`'anking'` vs `'classic'`).
2. AnKing IOE expects its model GUID to be the canonical IOE GUID for "siblings" / "edit cards" actions to flow through the add-on. Do we use the canonical IOE model ID (`1374772155` from AnkiWeb) or a stable derived ID? Wrong choice = users can edit cards but the add-on's review hooks won't fire on our notes.
3. AnKing's published note type ships with specific CSS variable names for mask colours (`--inactive-shape-color`, etc.). We already use a similar set — confirm the exact names match AnKing's current build, not our 2024 set.
4. Should we expose `Footer` / `Remarks` / `Sources` / `Extra 1` / `Extra 2` as optional inputs on the IO screen, or always emit them empty? Recommendation: ship empty in v1 (matches today's UX), expose in a follow-up if real users ask.
5. Does `ApkgPreviewService` already render AnKing-style SVG-overlay IO cards, or does the existing fallback ("Image Occlusion cards open with masks in Anki") still apply? If preview support is missing for AnKing-shaped templates, file a follow-up issue but do not block this spec.

**Out of scope (next iteration)**:
- Auto-suggest of occlusion regions (PR #2614).
- AnKing Step Decks note-type compatibility.
- A "rewrite my existing 2anki IO deck into AnKing shape" import-and-convert flow.
- Exposing the extra AnKing fields (`Footer`, `Remarks`, `Sources`, `Extra 1`, `Extra 2`) in the IO upload UI.

## Design notes

User moment: a returning student lands on `/image-occlusion`, drops in 4 lecture slides, expects the deck they download to live next to their AnKing IOE cards.

Recommendation: add a single segmented control under the deck-name input, **before** the upload area. Label "Note type". Two segments: `2anki classic` and `AnKing-compatible`. Selected state matches the existing segmented controls on the upload page (per `style/spec-selected-state-polish`). One-line helper below the control:

> AnKing-compatible matches the Image Occlusion Enhanced note type, so cards merge into AnKing decks. Pick 2anki classic if your existing 2anki cards already use it.

Primary action stays `Make deck`. No new screen, no modal, no "are you sure?" — the toggle is the whole UX.

Empty state for the helper line is identical regardless of selection; we don't need a copy variant.

Error states (no images, image too large, free-tier exceeded) are unchanged — they fire from the same use case.

Copy strings:
- Toggle label: `Note type`
- Segment 1: `2anki classic`
- Segment 2: `AnKing-compatible`
- Helper text: `AnKing-compatible matches the Image Occlusion Enhanced note type, so cards merge into AnKing decks.`
- (No new error strings — existing IO errors carry across both modes.)

Verdict: minor UI surface only; no PM-scope changes required.

## Technical pre-flight

**Layers touched**:
- `web/` — IO upload page: add segmented control, send `noteType` in the multipart payload, persist user preference.
- `routes` — `ImageOcclusionRouter.ts`: no change to route shape; existing `data` JSON field carries the new key.
- `controllers` — `ImageOcclusionController.ts`: parse `noteType` from `RawData`, default to `'anking'`, pass through.
- `usecases` — `CreateImageOcclusionDeckUseCase.ts`: accept `noteType` on input, forward into `deck_info.json`.
- `services` — `officialTemplates.ts`: register the new template so the existing template-listing API surfaces it; preview path may need a fallback (see open question 5).
- `data_layer` — only if open question 1 lands on "new column / new table". Otherwise no DB work.
- Python bridge — `create_deck/create_io_deck.py`, `create_deck/helpers/get_model.py`, `create_deck/helpers/io_shapes.py`, plus a new `helpers/io_shapes_to_svg.py` for the AnKing converter.

**Files likely in play**:
- New: `src/templates/n2a-io-anking.json` — AnKing field shape + card template.
- New: `create_deck/helpers/io_shapes_to_svg.py` + colocated `tests/test_io_shapes_to_svg.py`.
- Edit: `src/usecases/imageOcclusion/CreateImageOcclusionDeckUseCase.ts` + test.
- Edit: `src/controllers/ImageOcclusionController.ts`.
- Edit: `create_deck/create_io_deck.py` (branch on `noteType`).
- Edit: `create_deck/helpers/get_model.py` (register `"io-anking"`).
- Edit: `src/services/officialTemplates.ts` + test (new entry `official-n2a-io-anking`).
- Edit: web IO page component(s) — add the segmented control + payload field.
- New: `web/src/pages/WhatsNewPage/changelog/YYYY-MM-DD-image-occlusion-anking-compatible.json` at implementation time.

**Cross-language coordination**: TypeScript ↔ Python contract widens by one field (`noteType: 'classic' | 'anking'`) in `deck_info.json`. The Python side branches on it; default if missing is `'classic'` so the existing call shape stays valid through any rollout glitch.

**Estimated effort**: **M (Medium)**. The shapes-to-SVG converter (rect / ellipse / polygon with labels) is the load-bearing new code — maybe 80 LOC plus tests. Everything else is wiring through layers we already own. The 11-field note type is a JSON file; the AnKing template HTML is a one-time port. No new external dependencies.

**Security / testing / migration concerns to address before work starts**:
- *Security*: SVG strings are reflected into card HTML at render time inside Anki, not in our server's response. Still: when building the SVG, never inject the user-supplied `label` field unsanitised. The label can be empty today, but the converter is the right place to enforce length cap + HTML-escape for the inevitable future case where it carries user text. CWE-79.
- *Testing*: outside-in test on `CreateImageOcclusionDeckUseCase` with `noteType: 'anking'`, asserting the workspace's `deck_info.json` carries the new key. Colocated Python test for shape-to-SVG conversions covering rect, ellipse, polygon, and a label with `<`/`&`/quotes. Snapshot test of the registered template entry in `officialTemplates.ts`. No new external mocks needed.
- *Migration*: only if open question 1 resolves toward a new column / table. If yes, write `<n>_add_image_occlusion_note_type_preference.js` plus the corresponding `pnpm kanel` regen of `src/data_layer/public/`. Do not edit `data_layer/public/` by hand.
- *Sonar*: the new converter likely has a `switch` over shape kind. Keep each branch flat — Sonar S3776 (cognitive complexity) bites at nesting depth 4+. Extract per-shape builders.
- *Preview*: confirm `ApkgPreviewService` either renders SVG-overlay templates or falls back gracefully. If preview goes blank for AnKing-shape decks, the user lands on a broken upload-result screen — non-trivial regression risk for an opt-in toggle. Flag as follow-up if it's not free.
