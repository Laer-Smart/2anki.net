# Spec: Visual assets for the cards docs pages

Issue: #2468 (follow-up from #2463, #2464, #2467)

## Problem

Two docs pages ship prose-only: **Image occlusion** (`/documentation/cards/image-occlusion`) and **Note types and templates** (`/documentation/cards/templates`). The image-occlusion canvas tool is inherently spatial; the templates library is inherently visual. Without a short demo clip and a screenshot, both pages undersell features that sit top-of-funnel for the 300K-user goal — a first-time visitor reading text cannot picture drawing occlusion boxes or browsing the template grid.

## Proposal

Add three visual assets and wire them into the two docs files. The asset specs in the issue are exact and authoritative — follow them verbatim:

1. **Image-occlusion demo** — `web/public/docs-assets/image_occlusion_demo.mp4`, 8–12 s, 100% zoom, canvas-cropped, encoded under 2 MB (`ffmpeg ... -crf 28 -preset slow -an`). mp4 with `autoplay loop muted`, not GIF (size).
2. **Templates grid screenshot** — `web/public/docs-assets/templates_grid.webp`, 16:9, 1280px 1x, WebP q80, under 150 KB. "Your note types" empty is fine.
3. **YouTube walkthrough** — 3–4 min, silent with burned-in captions per VOICE.md (specific, sentence case, no fake warmth, no exclamation marks). Sections: image occlusion (~90 s), templates (~60 s), hide-all vs hide-one (~60 s). Uploaded unlisted/public, link pasted in the implementation PR body.

Wiring (markdown snippets are spelled out in the issue):
- `cards/image-occlusion.md` — embed the mp4 (raw `<video>` HTML) just before step 4 of "Build a deck", caption beneath it, then the YouTube `<figure><iframe>` directly below.
- `cards/templates.md` — embed the grid screenshot at the top of "Browse the library" (before the bullet list), plus a single plain-text link to the walkthrough. No iframe on the reference page — the iframe belongs only on the workflow (image-occlusion) page.

The doc loader (`web/src/pages/DocsPage/loader.ts`) rewrites `](../assets/foo.png)` → `](/docs-assets/foo.png)`; raw HTML is already accepted in docs (`start-here/upload-a-file.md` is the precedent for both the `<video>` and `<iframe>` patterns).

## Media-production dependency (the real blocker)

This work cannot be completed by code edits alone. The three assets require human media production:

- Screen-recording the live image-occlusion tool at 100% zoom and encoding to mp4 under budget.
- Capturing and exporting the templates page as a WebP under 150 KB.
- Recording, captioning (burned-in, not auto-generated), and uploading a 3–4 min YouTube walkthrough.

`/implement` **cannot start until the three asset files exist** (mp4 + webp committed, YouTube link in hand). The markdown wiring is ~20 minutes of work; the assets are the long pole and depend on Alexander (or whoever owns the recording) producing them. Treat this spec as blocked-on-media until the files land in `web/public/docs-assets/`. To unblock incrementally, the markdown can be wired per-asset as each lands — but the page should never reference an asset that isn't committed (a broken `<video>`/`<img>` is worse than prose-only).

## Scope

- `web/public/docs-assets/image_occlusion_demo.mp4` (new, under 2 MB)
- `web/public/docs-assets/templates_grid.webp` (new, under 150 KB)
- `web/src/pages/DocsPage/content/cards/image-occlusion.md` — mp4 + iframe embed
- `web/src/pages/DocsPage/content/cards/templates.md` — screenshot + plain-text video link
- YouTube upload (external; link recorded in PR body)

## Explicitly NOT in scope

- No changelog entry — adds visuals to existing docs, no new capability (issue says so).
- No trio review for the implementation PR — docs-only additive content is trio-optional per CLAUDE.md.
- No new doc loader features — the existing rewrite + raw-HTML support already cover the embeds.
- GIF fallback — mp4 is the decision; do not also ship a heavy GIF.
- Captions auto-generated by YouTube — burned-in only, so they render in the silent embed.

## Touch points

- `web/public/docs-assets/` (new binary assets)
- `web/src/pages/DocsPage/content/cards/image-occlusion.md`, `.../templates.md` (web docs content)
- `web/src/pages/DocsPage/loader.ts` (read-only — relied on, not modified)

## Risks / Rails

- **Media production is the dependency, not the code.** The PR is gated on three human-produced assets; without them the wiring is meaningless. State the blocker plainly and do not merge markdown that points at uncommitted files.
- **Asset size budgets** are correctness criteria, not nice-to-haves — an 8 MB "demo" mp4 regresses page load on a top-of-funnel page. Verify each file is under budget before commit.
- **Browser check applies** — the PR touches `web/src/`; the attestation gate fires. Verify the mp4 autoplays muted, the iframe loads, and there are no console errors at 375 px before merge.
- No auth / payments / migration / integration surface.

## Acceptance criteria

- `image_occlusion_demo.mp4` committed, under 2 MB; `templates_grid.webp` committed, under 150 KB.
- YouTube walkthrough uploaded (unlisted or public), link in the PR body.
- `cards/image-occlusion.md` renders the mp4 (autoplay, loop, muted, playsinline) with the caption and the YouTube iframe below it.
- `cards/templates.md` renders the grid screenshot at the top of "Browse the library" and a plain-text link to the walkthrough — no iframe.
- No page references an asset that is not committed. `/check` and CI green. Browser-check attestation in the PR body.
