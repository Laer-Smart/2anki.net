# Cascade user-delete to the email-tracking tables

**Issue:** none — surfaced from prod error log 2026-05-21
**Status:** draft — ready for `/implement`

---

## Problem

User deletion fails when the account has been sent an inactivity email. Today's prod error log:

```
delete from "users" where "id" = $1 violates fk "inactivity_emails_user_id_foreign"
detail: Key (id)=(2937) is still referenced from table "inactivity_emails".
```

`UsersRepository.deleteUser` (`src/data_layer/UsersRepository.ts:94-113`) sweeps a hard-coded list of ten owner tables before deleting from `users`. Two newer email-tracking tables are missing from the list **and** declare their FK to `users(id)` without `ON DELETE CASCADE`:

| Migration | Table | FK | Live rows |
|---|---|---|---|
| `20260526000000_inactivity_emails.js:4` | `inactivity_emails.user_id` | no `onDelete` | 7 000 |
| `20260520000000_re_engagement_emails.js:4` | `re_engagement_emails.user_id` | no `onDelete` | 0 (table just rolled out) |

Result: every user who has been sent an inactivity email since 2026-05-13 cannot be deleted — by the reaper, by an account-cleanup request, or by any future GDPR flow. User 2937 is the confirmed case in the log; there are 7 000 such rows today and the count grows every batch.

---

## Audience

Internal. Affects the user-deletion path used by the inactivity reaper and any GDPR-style account-delete request.

---

## Scope — v1 in

- One migration that re-adds both FKs with `ON DELETE CASCADE`:
  - `inactivity_emails.user_id` → `users(id) ON DELETE CASCADE`
  - `re_engagement_emails.user_id` → `users(id) ON DELETE CASCADE`
- Down migration restores the prior (no-cascade) shape so the migration is reversible.
- `pnpm kanel` to regenerate `public/InactivityEmails.ts` and `public/ReEngagementEmails.ts`.
- One integration test in `UsersRepository.test.ts`: insert a user, seed one row in each of the two email tables, call `deleteUser`, assert all three rows are gone.

## Scope — v1 out

- Auditing every FK to `users(id)`. `favorites` is the only other no-`onDelete` FK left, but `deleteUser` already sweeps it manually — no live breakage. Defer to a separate audit.
- Backfill or repair of orphan rows that already exist. The migration fixes the schema; the next delete attempt against any blocked user will succeed because CASCADE removes the dependent rows.
- Removing the `ownerTables` array from `deleteUser`. That's a separate refactor — see "Why CASCADE, not manual sweep" below.

## Why CASCADE, not manual sweep

`deleteUser`'s hard-coded `ownerTables` list is a maintenance hazard. Every new user-scoped table becomes one more thing to remember. CASCADE puts the contract in the schema where it belongs and removes the foot-gun for the next `notification_*` table someone adds. Other user-FK tables in the codebase already use `onDelete('CASCADE')` — `magic_tokens`, `io_drafts`, `chat_messages`, `conversations`, `user_passes`, `deck_shares`. The two email tables are the outliers.

A follow-up refactor can drop the now-redundant entries from `ownerTables`, but that's not load-bearing for this fix.
